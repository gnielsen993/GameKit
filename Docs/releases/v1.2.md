# v1.2

Date: in progress (2026-05-12 → )
Type: minor

## Summary
- Video Mode milestone — optional layout adaptation that keeps GameDrawer
  playable while a floating PiP video sits on screen. User manually picks
  one of six PiP positions (Large top/bottom + 4 corners); every game
  screen reflows accordingly. Design-first build: Phase 8 locked the
  visual language against real screenshots before any code shipped.

## User-facing changes
- **Phase 9 (Video Mode foundation).** Settings → VIDEO MODE card with
  an Off/On toggle (default Off, persisted across launches). When On,
  a "Video location" link reveals a visual iPhone-outline picker — tap
  any of 6 zones (Large top, Large bottom, Small top-left / top-right /
  bottom-left / bottom-right) to select where your video appears on
  screen. The selected zone highlights with the active theme's accent
  color and shows "Your video will go here". A short paragraph below
  the picker explains that GameDrawer can't detect your video
  automatically — the manual pick keeps the board and controls clear.
- No game layout changes in this phase. Mines, Merge, and Nonogram
  render byte-identical to v1.1 (Phase 10–12 introduce per-game
  reflow).
- **Phase 10 (Video Mode layout primitives).** No user-facing change.
  Internal-only — ships the `.videoModeAware()` modifier + slot-router
  helper consumed by Phase 11+. Mines / Merge / Nonogram still render
  byte-identical to v1.1 with Video Mode toggled either Off (default)
  or On.

## Internal changes
- **VideoModeStore** (`Core/VideoModeStore.swift`) — @Observable
  @MainActor user-preference store mirroring the Phase 4 SettingsStore
  pattern verbatim; backs `isEnabled` + `location` to UserDefaults.
- **VideoModeLocation** enum (`Core/VideoModeLocation.swift`) — 6-case
  raw-string-stable vocabulary.
- **VideoCompactControlRow** (`Core/VideoCompactControlRow.swift`) —
  generic `@ViewBuilder` shared compact control row component
  (Back | primary info | picker | secondary info | settings); reads
  Phase 8 token anchors (`theme.radii.button`, `theme.spacing.xl`,
  `theme.spacing.s`); #Preview block shows the 3 game slot mappings
  (Mines / Merge / Nonogram) — Phase 11/12 will adopt it.
- **VideoLocationPickerView** (`Screens/VideoMode/`) — visual iPhone-
  outline picker using GeometryReader + 9:19.5 aspect ratio; 6 zones
  laid out per Phase 8 VIDEO-MODE-LAYOUTS.md vocabulary; A11Y per zone
  (label + isButton trait + selection value).
- **GameKitApp.swift** — adds the VideoModeStore to the 5-store
  injection chain (no impact on existing 4 stores).
- **Localizable.xcstrings** — 12 new `videoMode.*` keys including the
  VIDEO-14 verbatim manual-selection copy.
- **Tests** — 7 new test files (14 @Test funcs) covering store round-
  trip, environment injection, Settings binding, picker tap+a11y,
  catalog key existence, and the SC5 off-state contract.
- **VideoModeAware** (`Core/VideoModeAware.swift`) — `ViewModifier`
  conformance shipping the v1.2 layout contract:
  - Off-path hard short-circuit (`if !store.isEnabled { return AnyView(content) }`)
    — byte-identical to un-wrapped view per CONTEXT D-05 / VIDEO-13.
  - Large-PiP band reservation via `.safeAreaInset(edge: .top/.bottom)`
    at `geometry.size.height * 0.32` (measured from
    `Docs/screenshots/v1.2-design/home-classic-pip-large-bottom.png`,
    `private static let largeBandFraction` per D-09 / D-10).
  - Compactness publication via custom `EnvironmentKey`
    (`\.videoModeCompactness` — 3 levels: `.normal` / `.collapsedSettings`
    / `.reducedTime`) computed from available board height vs
    `minBoardHeight` parameter per D-13 / D-14 0.85× threshold.
  - 12 named `#Preview` blocks (6 PiP zones × Classic Chrome Diner +
    Dracula) for SC5 manual legibility audit per D-16.
- **VideoModeSlotRouter** (`Core/VideoModeSlotRouter.swift`) — pure
  Foundation-only helper. `static func anchors(for: VideoModeLocation)
  -> SlotAnchorMap` returns the 4-slot anchor map (back / settings /
  picker / fab) per the locked switch table — data sourced from Phase 8
  `VIDEO-MODE-LAYOUTS.md`. Phase 11/12 game views read this in their
  slot reposition logic (P10 ships zero adoption).
- **Adoption API** — `View.videoModeAware(minBoardHeight: CGFloat = 320)`
  extension. One-line call site for every Phase 11/12 game view per
  CONTEXT D-01 (wraps the game view at the outermost layer; D-15 keeps
  `MinesweeperBoardView`'s `MagnifyGesture` + auto-scale stack
  UNTOUCHED).
- **Tests (Phase 10)** — 2 new test files: `VideoModeAwareTests.swift`
  (4 @Test funcs covering off-state byte-identical + 3 compactness

  in commits `fcdbafd` / `7e97b7e`; GREEN flip in `c1c5830` (router) +
  `e2bbc18` (modifier).

## Risks / notes
- VideoMode is **Off by default** and no game view reads the store in
  Phase 9, so the off-path is automatic byte-equivalence with v1.1
  (CONTEXT D-15). Phase 11/12 introduce the actual reads.
- Theme audit on the picker sub-screen confirmed legible on Classic
  (Chrome Diner) + at least one Loud preset (Voltage or Dracula) per
  CLAUDE.md §8.12. The selected-zone fill uses
  `theme.colors.accentPrimary.opacity(0.25)` with the "Your video will
  go here" label set to `theme.colors.textPrimary` — pre-empts the
  accent-on-accent legibility failure mode flagged in Phase 9
  research.
- MARKETING_VERSION remains 1.1; bumped to 1.2 in a later Phase 13
  ship plan.
- **`AnyView` short-circuit cost (off-path, Phase 10)** — accepted per
  CONTEXT D-05. Toggling Video Mode on→off triggers a clean SwiftUI
  rebuild of the wrapped game view (no leftover insets, no env values).
  If Phase 11 reveals a state-loss regression (e.g., timer reset on
  mid-game toggle), the remediation is to promote state UP one level
  above the modifier, NOT to eliminate the AnyView short-circuit (the
  byte-identical-off-restore guarantee is the SC3 contract).
- **Hard-Mines `MagnifyGesture` UNTOUCHED (CONTEXT D-15, Phase 10)** —
  Phase 10 wraps `MinesweeperGameView` (not `MinesweeperBoardView`) in
  Phase 11 adoption. The 06.1-03 / A11Y-05 `MagnifyGesture` + auto-scale
  `cellSize(forWidth:...)` stack stays byte-identical. The only
  Video-Mode-aware seam in Mines is `MinesweeperBoardView.Self.minCellSize`,
  gated on `videoModeStore.isEnabled` in Phase 11 per
  `08-HARD-MINES-ADR.md` (smaller-cells / Variant 1).
- **Screenshot README device-label drift (Pitfall 5, Phase 10)** —
  `Docs/screenshots/v1.2-design/README.md` claims iPhone 17 Pro Max but
  pixel dimensions are 17 Pro / 16 Pro (1179×2556px = 393×852pt). The
  `largeBandFraction = 0.32` measurement is device-portable (it's a
  fraction of `geometry.size.height`), so this has no code impact — but
  the README annotation can be corrected in a future docs commit.
- **SC5 audit-helper iteration (Phase 10)** — Plan 10-03 initial preview
  helpers had two audit blockers (Dracula resolved in light scheme; no
  visual PiP indicator on Small zones). Fixed in Plan 10-04 follow-up
  (commit `9bc50ab`) — preview-only changes, no production logic
  touched. Pattern (translucent corner box / edge band labelled "PiP")
  is reusable for the Phase 13 win/loss banner audit.
