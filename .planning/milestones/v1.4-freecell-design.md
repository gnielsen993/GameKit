# FreeCell — Implementation Design
**Status:** Planning — not yet started  
**Part of:** v1.4 milestone (FreeCell first, Klondike second)  
**Created:** 2026-05-23

---

## 1. Rules

### Win condition
All 52 cards on the 4 foundations, built A → K by suit.

### Loss condition
No legal move exists: all 4 free cells occupied, no column card can stack anywhere, no card can advance to foundation. Rare (~0.1% of non-excluded deals). Must be detected and surfaced as a game-over state.

### Move rules

**To a column:**
Card can land on a column top if it is one rank lower and opposite color.
Empty column: any card (or legal sequence) may go there.

**To a free cell:**
Any single card. A free cell holds exactly one card.

**To foundation:**
The next card in suit sequence (A first, then 2, 3, … K).

**Supermove (sequence moves):**
A contiguous sequence of alternating-color, descending-rank cards may be moved in a single gesture if there are enough staging positions. Formula:

```
max_moveable = (empty_free_cells + 1) × 2^(empty_columns)
```

If the destination is an empty column, that column does not count as an empty column for the formula:

```
max_moveable = (empty_free_cells + 1) × 2^(empty_columns − 1)
```

Implement the supermove check on gesture start — if the picked-up sequence is too large for the current board state, reject immediately (error haptic, no pickup).

## 2. Board Layout

### Structure
```
[ FC0 ] [ FC1 ] [ FC2 ] [ FC3 ]    [ F♠ ] [ F♥ ] [ F♦ ] [ F♣ ]
─────────────────────────────────────────────────────────────────
[ col0 ] [ col1 ] [ col2 ] [ col3 ] [ col4 ] [ col5 ] [ col6 ] [ col7 ]
```

Top row: 4 free cells (left) + 4 foundations (right).  
Columns: 8, fill remaining height with `layoutPriority(1)`.

### Top row visual language
- Free cell slot: dashed/dotted rounded rect outline when empty. Shows card when occupied.
- Foundation slot: suit symbol centered in the slot when empty. Shows top card when occupied (A, then building up).
- Gap or thin divider line between free-cell group and foundation group.

### Card sizing (scales to device)

Cards fill available column width. 8 columns + 7 gaps on usable board width:

| Device | Screen width | H-pad × 2 | Usable | Gap | Card width | Card height |
|--------|-------------|-----------|--------|-----|------------|-------------|
| iPhone SE | 320 | 24 | 296 | 4 | 33.5 | 46.9 |
| iPhone 16 | 390 | 24 | 366 | 4 | 41.4 | 58.0 |
| iPhone 16 Pro Max | 430 | 24 | 406 | 4 | 45.7 | 64.0 |

Fan overlap (vertical offset between cards in a column): `cardHeight × 0.30` — large enough to show rank + suit corner but keep columns compact.

Minimum readable card size: 33pt wide / 46pt tall (iPhone SE baseline). If this becomes unreadable on some device, add horizontal scroll to the column area (but avoid this if possible — standard FreeCell never scrolls).

### Column depth
Columns 0–3 start with 7 cards; columns 4–7 start with 6 cards (standard FreeCell deal — 52 cards / 8 columns = 6.5 → first 4 get the extra card).

Maximum column depth during play: theoretically up to 18+ cards if the user stacks badly. The column must scroll vertically or compress fan overlap when it grows beyond the available height. Simplest approach: reduce `fanOffset` proportionally as card count grows past 8, with a minimum fanOffset floor of `cardHeight × 0.18`.

## 3. Deal Algorithm

### Deal numbering
Use the classic Microsoft FreeCell RNG — deals are fully deterministic from a seed (deal number). This allows deal sharing ("play deal #617") and excludes known-unsolvable deals by blacklist.

### Shuffle algorithm (LCG)
```swift
func microsoftShuffle(dealNumber: Int) -> [PlayingCard] {
    var state = dealNumber
    var deck = standardDeck()  // 52 cards, ordered
    var result: [PlayingCard] = []

    for remaining in stride(from: 52, through: 1, by: -1) {
        state = (state &* 214013 &+ 2531011) & 0x7FFF_FFFF
        let index = (state >> 16) % remaining
        result.append(deck.remove(at: index))
    }
    return result
}
```

Use `&*` and `&+` (overflow operators) because this algorithm intentionally wraps at 32-bit boundaries.

### Deal range
Ship deals #1 – #32,000 (the classic Microsoft range). Random mode picks a random number from this range.

### Known-unsolvable exclusions
Source a complete list before shipping. Currently documented unsolvable deals in range 1–32,000: **#11982** and **#146,692** (outside range). Additional unsolvable deals exist — reference the FreeCell FAQ maintained by Michael Keller / Solitaire Laboratory for the definitive exclusion list. Exclude from random selection and show a clear error if the user enters an excluded deal number directly.

### How cards are dealt to columns
Take the shuffled 52-card array and distribute round-robin:
- Card 0 → column 0, card 1 → column 1, …, card 7 → column 7
- Card 8 → column 0, card 9 → column 1, …
- Columns 0–3 end up with 7 cards; columns 4–7 get 6 cards.

## 4. Game Modes (Drawer chips)

Five chips. All play identical FreeCell rules — same unlimited undo, same stats model.

| Chip | Behavior |
|------|----------|
| **Easy** | Random deal from the Easy bucket of the pre-computed difficulty index. |
| **Medium** | Random deal from the Medium bucket. |
| **Hard** | Random deal from the Hard bucket. |
| **Expert** | Random deal from the Expert bucket. |
| **Deal #** | Launches game, immediately shows a number-entry sheet (1–32,000). User can cancel back to home. Shows "Your best: X:XX" if the deal has been played before. |

### Difficulty index
A bundled data file maps every deal #1–32,000 to a difficulty tier. Generated offline by running a solver against all deals and classifying by solution depth / state-space size. Ships as a compact binary asset (~32 KB uncompressed). App reads it once at launch and keeps it in memory.

### Architecture — one code path for all modes
```swift
enum FreeCellMode {
    case easy, medium, hard, expert
    case dealNumber(Int)
}
// Daily challenge when it ships = .dealNumber(DailyChallenge.today())
// DailyChallenge.today() = validDeals[daysSinceEpoch % validDeals.count]
// No backend needed — date-derived, fully deterministic.
```

The game engine receives a deal number regardless of how it was launched. Difficulty chips just pick from the right bucket. Deal # passes the number directly. Daily challenge (future) passes a date-derived number through the same path.

## 6. Stats

Two SwiftData models. FreeCell and Klondike tracked separately.

### Global stats — one record, always exists
```swift
@Model class FreeCellStats {
    var gamesPlayed:   Int           // incremented on first move
    var gamesWon:      Int
    var currentStreak: Int           // resets on loss or abandon
    var bestStreak:    Int
    var fastestWin:    TimeInterval? // nil until first win
    var totalWinTime:  TimeInterval  // sum of all winning times
}
```
Win rate (`gamesWon / gamesPlayed`) and average time (`totalWinTime / gamesWon`) are computed, not stored.

### Per-deal stats — lazy, one record per played deal
```swift
@Model class FreeCellDealRecord {
    var dealNumber: Int
    var bestTime:   TimeInterval? // nil if never won
    var solved:     Bool
}
```
Records are created only when a deal is actually played — never pre-allocated. A user who plays 200 deals has 200 records (~20 KB total). Zero overhead for unplayed deals.

Querying by `dealNumber` is all daily challenge will ever need when it ships — no additional fields required now.

### Rules
- A game counts as "played" on the first move, not on deal.
- Restarting the same deal does not add another "played" count.
- No per-difficulty stat split — fragments numbers for no real user benefit.

### User-visible display
Stats screen: Played · Won · Win % · Streak · Best Time · Avg Time.
After winning Deal #N: "Your best: 3:42" shown inline — enough for personal competition and sharing.

## 7. Undo System

Unlimited undo — this is expected by FreeCell players and the game is a planning exercise, not a reflex game.

### Move model
```swift
enum CardSource {
    case column(Int)
    case freeCell(Int)
}

enum CardDestination {
    case column(Int)
    case freeCell(Int)
    case foundation(Int)
}

struct FreeMove {
    let cards: [PlayingCard]   // 1 card for single moves, N for supermoves
    let source: CardSource
    let destination: CardDestination
    let timestamp: Date        // for time tracking
}
```

Store a `[FreeMove]` history stack. Undo pops the last move and reverses it. Redo: not needed (undo-only is standard for solitaire). Auto-complete moves are added to history so they can be undone.

### Undo UI
Undo button in toolbar. Disable (grey out) when history is empty. No undo count badge needed.

## 8. Win Detection & Auto-complete

### Win detection
After every move: check if all 4 foundations have 13 cards. If yes, trigger win sequence.

### Auto-complete trigger
After every move: check if every remaining card is (a) on a column and (b) already in the correct order to cascade up to foundations without any column-to-column moves. If yes and auto-complete is enabled, begin auto-completing.

Auto-complete algorithm: repeatedly find the lowest-ranked unplaced card that can safely go to its foundation (safe = no card of opposite color needs it as a column target anymore), move it, repeat until done.

### Win sequence
1. Remaining cards animate off to their foundations in cascade order (low ranks first).
2. Win banner appears (same pattern as other games: `EndStateBannerView`).
3. Stats updated.
4. Options: New Game / Replay Deal / Done.

## 9. Interaction Model

### Primary: drag
- Touch and hold a card (or card sequence) and drag to destination.
- Supermove check on pickup: if picked-up sequence exceeds max_moveable, reject (error haptic, no pickup animation).
- Dragging an occupied free cell card picks it up as a single card.
- Drop target highlights: green border for valid, no highlight for invalid.
- Snap-back animation on invalid drop (same as tableau prototype).

### Secondary: tap-to-move
- Tap a card → it gets a selection highlight (accent color border/glow).
- While selected: tap any valid destination → card moves there.
- Tap a different card → switch selection to that card.
- Tap selected card again → deselect.
- Selection is cleared when a drag starts.

### Auto-move shortcuts
- **Double-tap** a card → move to foundation if legal.
- **Long-press** a card → same as double-tap (alternative for accessibility).

### Empty free cell tap
Tapping an empty free cell while a card is selected → moves selected card to that free cell.

## 10. Video Mode (PiP)

Follows DESIGN.md §7 rules. Board gets `layoutPriority(1)`.

### Normal layout (all zones)
Standard layout: top row (free cells + foundations) + 8 columns.

### Large PiP zone
Top row compresses:
- Free cell slots and foundation slots shrink to minimum tap target (44pt × 44pt area, card renders smaller inside).
- Reduce top row vertical padding.
- Column fan offset may also reduce slightly to reclaim vertical space.

### Small PiP zone (≤ 2 chips per DESIGN.md §7)
Free cells become overlay badges on the board corners, or collapse entirely to a compact strip. Foundation state shown as 4 small suit + rank indicators in a single row above the board. Columns fill all available height.

Design this layout before implementing — requires a sketch pass.

## 11. Animations

| Event | Animation |
|-------|-----------|
| Card pickup | Scale 1.04, lift shadow (same as tableau prototype) |
| Card move (tap-to-move) | Spring arc from source to destination, 0.28s |
| Snap-back (invalid) | Spring back to origin, error haptic |
| Valid drop | Spring settle, success haptic |
| Card to foundation | Slight upward arc then settle, lighter success haptic |
| Auto-complete sequence | Cards fly to foundations in rapid succession, 0.15s each, staggered 0.08s apart |
| Win banner | Fade + scale in after last auto-complete card lands |
| Undo | Reverse spring, card flies back to source |

All animations must respect Reduce Motion: replace spring arcs with simple cross-fades.

## 12. Engine Architecture

Pure Swift, no SwiftUI imports:

```
Games/Solitaire/
  Engine/
    FreeCell/
      FreeCellDeal.swift       — shuffle algorithm, deal struct
      FreeCellEngine.swift     — board state, move validation, undo stack
      FreeCellWinDetector.swift — win + auto-complete detection
      SupermoveCalculator.swift — supermove max calculation
  CardModel.swift              — shared (already exists)
```

`FreeCellEngine` is the single source of truth for board state. Views observe it and render. Views never mutate board state directly — all mutations go through engine methods that also record undo history.

## 13. Open Questions (resolve before starting implementation)

1. **Unsolvable deal list** — source the complete exclusion list for #1–32,000. The two cited ones may not be exhaustive.
2. **Column overflow strategy** — reduce fan overlap dynamically, or add a scroll container per column? Dynamic reduction is simpler and matches physical card games.
3. **Loss detection frequency** — check after every move (correct) or only when the user explicitly asks? Always check — if the game is lost, surface it immediately.
4. **"Played" counter trigger** — first move, or on deal? Settled above as first move. Confirm.
5. **PiP small zone top-row design** — needs a sketch before implementation. Can defer until Video Mode pass.
6. **Klondike data sharing** — FreeCell and Klondike share `CardModel.swift` and `PlayingCardView`. What else can be shared? `FreeCellEngine` and `KlondikeEngine` will be separate. Foundation/free cell view components may be shareable.
