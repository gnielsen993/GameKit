## Visual Contract ŌĆö Win vs Loss Differentiation

Banner shape is identical for win and loss; differentiation comes through:

| Surface | Win | Loss | Source |
|---------|-----|------|--------|
| Title text color | `theme.colors.success` | `theme.colors.danger` | Mirrors existing `EndStateCard` (MinesweeperEndStateCard.swift:42, MergeEndStateCard.swift:31) |
| Surface fill | `theme.colors.surface` (NOT tinted) | `theme.colors.surface` (NOT tinted) | Banner stays neutral ŌĆö color sits in the title only, keeps banner readable across all 6 presets |
| Primary CTA (`DKButton`) | `.primary` style (DesignKit default tint) | `.primary` style (DesignKit default tint) | One affordance, one style ŌĆö 08-BANNER-PLACEMENT D-11 |
| Glyph / icon | **None in v1.2.** Color-coded title carries the win/loss read. | **None in v1.2.** | Deferred per CONTEXT "Deferred Ideas" ŌĆö keeps banner footprint minimal |

**Rationale for color-on-title-only (no surface tint):** Tinted success/danger fills would force a contrast-pass on every preset (Voltage / Dracula / Bright). ## Visual Contract ŌĆö Per-Zone Anchor (D-09 LOCKED, opposite-of-PiP)

Verbatim from `08-BANNER-PLACEMENT.md` D-09. Downstream agents consume this table ŌĆö do not re-derive.

| PiP location  | Banner docks   | Vertical edge inset |
|---------------|----------------|---------------------|
| `largeTop`    | bottom edge    | `theme.spacing.m` above the compact control row |
| `largeBottom` | top edge       | `theme.spacing.m` below the compact control row |
| `smallTopLeft`  | bottom-right (bottom edge, anchored trailing) | `theme.spacing.m` from bottom safe-area + clear of bottom-trailing slot anchors |
| `smallTopRight` | bottom-left  (bottom edge, anchored leading)  | `theme.spacing.m` from bottom safe-area + clear of bottom-leading slot anchors |
| `smallBottomLeft`  | top-right (top edge, anchored trailing) | `theme.spacing.m` from top safe-area + clear of top-trailing slot anchors |
| `smallBottomRight` | top-left  (top edge, anchored leading)  | `theme.spacing.m` from top safe-area + clear of top-leading slot anchors |

**Width behavior on Small zones:** Banner still spans full-width-minus-`spacing.m`-margins (08-BANNER-PLACEMENT D-10 LOCKED ŌĆö pill is full-width). The "anchored leading / trailing" in the table refers to the banner's INTERNAL alignment of its content (title leading, CTA trailing on Large zones; mirrored on Small zones to stay clear of the covered PiP corner's vertical column). Banner does NOT side-shrink to a half-width pill on Small zones ŌĆö that would create asymmetric chrome reading under different PiP picks.

**Collision-with-chrome rule:** On Small zones the banner shares its anchor edge with chip-row chrome already routed by Phase 12.1 (`anchors.headerBar` / `anchors.picker`). The banner sits OUTSIDE the chip-row band ŌĆö i.e. closer to the safe-area edge than the chips ŌĆö so the chip row can be tapped or read at the same time as the banner. Sequencing rule: chips occupy the top/bottom 64ŌĆō80pt band per zone; the banner occupies an additional ~56pt band immediately further from the covered corner.

## Typography

Banner consumes the existing DesignKit semantic typography tokens ŌĆö no new tokens. Mirrors the existing `EndStateCard` font choices so VoiceOver order and visual hierarchy match the off-path surface.

| Role | DesignKit token | Used for |
|------|-----------------|----------|
| Title | `theme.typography.title` | "You won!" / "Bad luck" / "2048!" / "No moves" / "Solved" / "Out of lives" |
| Title (Nonogram puzzle title appended on win) | `theme.typography.body` | The picture-title reveal text (D-13-COPY for Nonogram). Rendered on the line below the headline, smaller. |
| CTA label | (whatever `DKButton(.primary)` already declares ŌĆö do not override) | "Restart" / "Continue" / "New puzzle" / "Try again" |

**Title vs titleLarge:** Existing `EndStateCard` uses `theme.typography.titleLarge`. Banner uses `theme.typography.title` (one step smaller) because the banner is a slim pill, NOT a full card ŌĆö `titleLarge` would force a taller pill that visually competes with the board. The smaller `title` keeps the pill at minimum height while still reading as a clear outcome statement.

**Weight + line-height:** Inherited from DesignKit token definitions ŌĆö do NOT override. CLAUDE.md ┬¦2 forbids reaching past semantic tokens.

## Component API Decisions (Claude's Discretion ŌĆö closed at UI-SPEC time)

### C-01 ŌĆö Shared vs per-game banner views

**Decision:** Ship a SHARED `VideoModeBanner` view in `Core/`. Per-game banner views are NOT created.

**Location:** `gamekit/gamekit/Core/VideoModeBanner.swift`

**Rationale:**
- CLAUDE.md ┬¦2 promotion threshold = "used in 2+ games". Banner is consumed by 3 games (Mines / Merge / Nonogram) ŌĆö clearly past the threshold.
- The 6-zone anchor table is shared, the pill shape is shared, the animation gating is shared. Forking into 3 per-game files would duplicate 80%+ of the surface and create the same drift defect that caused Phase 12.1 (per-game silent omissions in shared routing).
- Per-game divergences (Nonogram's appended puzzle title; Merge's "Continue" semantic) are content-level, not chrome-level ŌĆö solved by the C-02 content shape below, not by forking the view.
- File Ōēż500 lines per CLAUDE.md ┬¦8.5 ŌĆö banner view is ~150 LOC max (single VStack with title row + CTA), well under cap.

### C-02 ŌĆö `BannerContent` struct vs `@ViewBuilder` slots

**Decision:** Use a `BannerContent` struct (PoD). NO `@ViewBuilder` slot API.

**Shape:**
```swift
struct VideoModeBannerContent {
    enum Outcome { case win, loss }

    let outcome: Outcome
    let title: String                       // Mines: "You won!" / Merge: "You reached 2048!" / Nonogram: "Solved"
    let subtitle: String?                   // Nonogram appends puzzle title here on win; nil for Mines/Merge
    let primaryButtonLabel: String          // "Restart" / "Continue" / "New puzzle" / "Try again"
    let onPrimary: () -> Void               // Restart / Continue / NewPuzzle / TryAgain action

    // a11y
    let accessibilityLabel: String          // Pre-composed by the per-game adopter (existing EndStateCards already build this string)
}
```

**Rationale:**
- The banner's content surface is bounded and well-understood (title + optional subtitle + single CTA). A `@ViewBuilder` slot API would invite per-game customization the design contract explicitly forbids (banner is single-action; tinted surface is forbidden; secondary actions are forbidden).
- Struct gives the planner a checklist of fields. Per-game adopters can't accidentally omit a field (compile error) and can't accidentally add an extra button (no slot exists).
- VoiceOver `accessibilityLabel` is pre-composed by the per-game adopter (each game's view model already builds the equivalent string for the existing `EndStateCard`) ŌĆö banner doesn't re-build it. This preserves the existing `formatElapsed` / `LossContext` plumbing in each game without forcing `VideoModeBanner` to know game-specific state.
- If a future game (v1.3+ Sudoku) needs a different banner content shape, extend the struct with new optional fields, not the API surface.

### C-03 ŌĆö Extend `SlotAnchorMap` with `banner: SlotAnchor` vs map inline at call site

**Decision:** DO NOT extend `SlotAnchorMap` with a `banner` field. Map inline at the banner's call site via a new `VideoModeBannerAnchor` helper (parallel to but separate from `SlotAnchorMap`).

**Rationale:**
- `SlotAnchor` is a 4-corner conceptual enum (`topLeading`, `topTrailing`, `bottomLeading`, `bottomTrailing`, `inCompactRow`, `hidden`). The banner anchor is conceptually different ŌĆö it's an EDGE anchor (top edge vs bottom edge), with leading/trailing being a content-alignment hint within that edge, not a corner anchor.
- Adding a `banner: SlotAnchor` field would force the banner to either (a) misuse the corner anchors to encode edge-anchoring (semantic drift ŌĆö future readers would be confused) or (b) bloat `SlotAnchor` with new cases (`bottomEdge`, `topEdge`, `bottomEdgeTrailing`, etc.) that no other slot uses. Both options pollute the existing `SlotAnchorMap` contract.
- The banner is a single new surface that uses ONE anchor (`bannerEdge: BannerEdge`). Encoding that in a small dedicated helper keeps the existing router uncluttered and gives the banner its own surface for future tweaks (e.g., a v1.3+ portrait-PiP support adding new banner edges).

**Helper shape:**
```swift
// gamekit/gamekit/Core/VideoModeBannerAnchor.swift
enum VideoModeBannerEdge: Sendable, Equatable {
    case top                // Banner docks to top safe-area edge
    case bottom             // Banner docks to bottom safe-area edge
}

enum VideoModeBannerAlignment: Sendable, Equatable {
    case leading            // Banner content (title + CTA) anchored leading; CTA trails
    case trailing           // Banner content anchored trailing; CTA leads
    case fullWidth          // Banner content spans full width (title leading, CTA trailing ŌĆö the Large-zone default)
}

struct VideoModeBannerAnchor: Sendable, Equatable {
    let edge: VideoModeBannerEdge
    let alignment: VideoModeBannerAlignment
}

enum VideoModeBannerRouter {
    static func anchor(for location: VideoModeLocation) -> VideoModeBannerAnchor {
        // Returns the row from the anchor table above. Exhaustive switch on VideoModeLocation ŌĆö
        // mirrors VideoModeSlotRouter's exhaustiveness invariant per Phase 12.1 D-01.
    }
}
```

This keeps the banner anchor logic compile-time-exhaustive (which was the C-03 winning property) without polluting `SlotAnchorMap` (which is the dimension we don't want ŌĆö banner is not a "slot", it's an overlay edge surface).

**Forward-compat with the old `VideoModeSlotRouter.swift:28-33` comment:** That comment ("If Phase 13 chooses to share, extend SlotAnchorMap with a `banner: SlotAnchor` field") was a forward-compat hint, not a lock. Phase 13 closes that open question with the dedicated helper approach above.
