<objective>
Close out the Phase 10 VERIFICATION.md A2 carry-forward — empirically
measure the real available board height for `MinesweeperGameView` mounted
inside its actual `NavigationStack` with Video Mode On + Large-top
selected. If the measured height is insufficient (Hard 16×30 cannot fit
between the reserved band, the nav-bar title, and the compact row at the
chosen floor from Plan 11-05), apply the documented `safeAreaInsets.top`
adjustment per `10-RESEARCH.md` Assumption A2.

**This plan is empirical, not speculative** (per CONTEXT D-16 + 10-VERIFICATION.md
§"Carry-forward to Phase 11 / 12"). The adjustment is applied ONLY if the
measurement shows it's needed. Pre-emptively widening the safe-area
inset risks regressing the off-path (`AnyView(content)` short-circuit
must stay short-circuited when Video Mode is Off).

Inputs locked by earlier plans:
- Plan 11-03 — three-way layout branch; wrap site at `.videoModeAware(minBoardHeight: 480)`.
- Plan 11-04 — Large-zone composition with compact row.
- Plan 11-05 — `minCellSizeVideoMode` locked floor.

Output: 0 or 1 code file modified (depending on the measurement outcome).
- **If A2 measurement passes** (board height sufficient): no code change;
  Plan 11-06-SUMMARY.md records the measurement and closes A2.
- **If A2 measurement fails** (board too compressed): apply the documented
  `safeAreaInsets.top` adjustment inside `VideoModeAware.body(content:)`'s
  Large-zone branch; re-measure; record the delta and final state in
  Plan 11-06-SUMMARY.md.

This plan stays empirical — it cannot pre-decide the answer.
</objective>
