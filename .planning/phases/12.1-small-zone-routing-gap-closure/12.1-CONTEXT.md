## Phase Boundary

This is a tightly-scoped gap-closure phase. It does NOT introduce new Video Mode capability. It closes one P11 carryforward defect that surfaced in Phase 12's manual-check audit (2026-05-13) and blocked SC1 + SC3 from passing.

**In scope:**
- Wire `anchors.picker` into `smallZoneToolbarContent` (or sibling layout) for all 3 adopter games (Minesweeper, Merge, Nonogram) so the per-game ModePill (`MinesweeperModePill` / `MergeModePill` / `NonogramModePill`) repositions away from the bottom-center default on Bottom L/R Small zones.
- Add a HeaderBar reposition seam so `MinesweeperHeaderBar` / `MergeHeaderBar` / `NonogramHeaderBar` repositions away from the top-center default on Top L/R Small zones (today no anchor concept exists for HeaderBar in `VideoModeSlotRouter`).
- Re-run the 12 affected Small-zone rows (4 zones × 3 games) manually on Classic + one Loud preset (Voltage or Dracula) per CLAUDE.md §8.12.

- Append Phase 12.1 entries to `Docs/releases/v1.2.md` per CLAUDE.md §0.3 + §8.14.

**Out of scope:**
- Large-zone (`largeTop` / `largeBottom`) path. Already PASSES manual audit on all 3 games. `compactRowComposed` + `largeZoneLayout` are NOT touched.
- Off-path layout (`videoModeStore.isEnabled == false`). Must remain byte-identical (VIDEO-13 anchor).
- New Video Mode features. Phase 13 (Win/Loss Banner + A11y Gating) is the next forward-progress phase.
- `VideoModeSlotRouter`'s exhaustive switch on `VideoModeLocation`. The router already returns the correct `anchors.picker` for all 6 zones; the call sites just don't consume it. (Note: extending `SlotAnchorMap` with a new `headerBar` field IS in scope per D-01 below.)

### Why the gap exists (root cause)
The P11 plan + P12 plans both asserted in plan text that the Small-zone branch was fully implemented per D-MS-01 / D-MG-01 / D-NG-01. The actual implementation only routed back + settings because at P11 implementation time the team focused on toolbar items (which are physical SwiftUI ToolbarItems with explicit placement) and overlooked the picker (which is rendered inline in the VStack via the per-game ModePill child) and the HeaderBar (rendered at the top of the VStack). The implementation drift was the same in P11 and got mechanically copied into P12's adopter files.

## Implementation Decisions (Locked)

### Router contract
- **D-01 (LOCKED):** Extend `SlotAnchorMap` in `Core/VideoModeSlotRouter.swift` with a new `headerBar: SlotAnchor` field. Reason: the named-fields shape gives compile-time exhaustiveness (per the existing comment block at lines 52–57) and the next consumer (Phase 13 banner) will likely want a parallel anchor. Inline computation at call sites would re-introduce the exact silent-omission failure mode that caused this gap. Update `VideoModeSlotRouterTests` for the new field.
- **D-02 (LOCKED):** `anchors.headerBar` values per zone:
  - `largeTop` → `.inCompactRow` (consolidated into compact row, same as picker)
  - `largeBottom` → `.inCompactRow`
  - `smallTopLeft` → `.bottomLeading` (push HeaderBar down + left, opposite of covered corner)
  - `smallTopRight` → `.bottomTrailing` (down + right)
  - `smallBottomLeft` → `.topLeading` (HeaderBar already at top — keep there, away from bottom-PiP; lean leading)
  - `smallBottomRight` → `.topTrailing` (lean trailing)
- **D-03 (LOCKED):** `anchors.picker` values are NOT changed — the router already returns correct values per the original P10 D-02 lock. This phase only adds CONSUMPTION of those values at the 3 game-view call sites.

### Call-site shape
- **D-04 (LOCKED):** Each `+VideoMode.swift` adds a new `smallZoneLayout` view (sibling of `existingLayout` / `largeZoneLayout`). The game `body`'s 3-way branch (`if !isEnabled / else if location.isLarge / else`) changes the `else` arm from `existingLayout` to `smallZoneLayout`. Off-path (`isEnabled == false`) keeps calling `existingLayout` — byte-identity preserved.
- **D-05 (LOCKED):** `smallZoneLayout` consumes BOTH `anchors.headerBar` and `anchors.picker`. The implementation MAY share sub-views with `existingLayout` (e.g., extract `headerBarContent` / `modePillContent` / `boardContent` as `@ViewBuilder` properties) but MUST NOT modify the off-path's call site. CLAUDE.md §8.5 ≤500-line cap applies — Mines file is already 485 LOC (close to cap); planner picks split strategy if needed.
- **D-06 (LOCKED):** `smallZoneToolbarContent` (existing function in each `+VideoMode.swift`) is preserved — it still routes back + settings toolbar items. No regression to existing toolbar wiring.

### Per-game positions when `anchors.headerBar` is a bottom-anchor
- **D-07 (LOCKED):** When `anchors.headerBar` resolves to `.bottomLeading` or `.bottomTrailing` (Top L/R Small zones), the HeaderBar renders in a sibling HStack at the bottom of the board area, BEFORE the ModePill row. Visual order top→bottom: Board → HeaderBar (bottom-aligned) → ModePill (only if `anchors.picker` is ALSO a bottom anchor; otherwise ModePill goes wherever `anchors.picker` resolves). Reason: keeps the existing semantics that HeaderBar shows game-state chips (Score/Best/Timer/Mines/Lives/Size) and ModePill shows mode-switch; user reads chips above the picker.
- **D-08 (LOCKED):** When `anchors.picker` resolves to a top-anchor (Top L/R Small zones — `.bottomTrailing` / `.bottomLeading` per router), ModePill renders at the bottom of the board area (its semantic home) and HeaderBar moves to bottom per D-07. Top of screen is left clear for the top-PiP overlay.
- **D-09 (LOCKED):** When `anchors.picker` resolves to a bottom-anchor (Bottom L/R Small zones — `.bottomTrailing` / `.bottomLeading` per router; the router actually returns `.bottomTrailing` for smallBottomLeft and `.bottomLeading` for smallBottomRight per the existing `VideoModeSlotRouter` switch), ModePill renders at the TOP of the board area (opposite of covered corner) and HeaderBar stays at top per D-02 (`anchors.headerBar` = `.topLeading` / `.topTrailing`). Bottom of screen is left clear for the bottom-PiP overlay.

### Per-game guardrails
- **D-MS-10 (LOCKED):** Minesweeper's existing pinch-zoom (`MagnifyGesture` + `clampZoomScale`) + auto-scale system from 06.1-03 + A11Y-05 MUST NOT be touched. `MinesweeperBoardView` constructor call site is byte-identical to `existingLayout` per the original D-MS-17 untouched contract. `smallZoneLayout` shares the same board sub-view.

### Off-path byte-identity

### Expected `smallZoneLayout` shape (Mines reference; same pattern for Merge + Nonogram)
```
@ViewBuilder
var smallZoneLayout: some View {
    let anchors = VideoModeSlotRouter.anchors(for: videoModeStore.location)
    ZStack {
        theme.colors.background.ignoresSafeArea()
        VStack(spacing: theme.spacing.m) {
            // HeaderBar position depends on anchors.headerBar:
            // - top anchor → render at top of VStack (current existingLayout position)
            // - bottom anchor → render at bottom (after board, before/after ModePill)
            if anchors.headerBar.isTopAnchor { headerBarContent }
            boardContent  // shared with existingLayout (byte-identical call site)
            if anchors.picker.isTopAnchor { modePillContent }
            if anchors.headerBar.isBottomAnchor { headerBarContent }
            if anchors.picker.isBottomAnchor { modePillContent }
        }
    }
}
```
(Pseudocode — planner picks exact shape. The `isTopAnchor` / `isBottomAnchor` helpers may be added to `SlotAnchor` extension or inlined.)
