## Implementation Decisions

### Issue 1 — Home cards 2-per-row grid (SHELL-05)

- **D-01:** **2-column LazyVGrid layout, square cards.** Replace the existing vertical list (`HomeView` lines 28-39) with a `LazyVGrid` of 2 columns. Cards are square (1:1 aspect via `.aspectRatio(1, contentMode: .fit)`). Reads `theme.spacing.m` between columns and rows. Uses the existing `DKCard` component for each tile (tokens preserved).
- **D-02:** **Two cards only on Home: Mines (enabled hero) + Upcoming (placeholder).** The 8 disabled placeholders (Merge / Word Grid / Solitaire / Sudoku / Nonogram / Flow / Pattern Memory / Chess Puzzles) move OUT of the Home grid into a tap-revealed list inside the Upcoming tile. Home shows exactly 2 tiles in 1 row (or stacked if 2-col layout requires both cards in row-1).
- **D-03:** **Side-by-side 2-col layout.** Mines + Upcoming sit in a single row, both square, both same size. NOT a hero+chip layout, NOT a stacked layout — two equal-weight tiles side-by-side per the user's verbatim "side-by-side 2-col" choice.
- **D-04:** **Upcoming tile opens a sheet listing 8 planned games.** Tap on Upcoming → present a `.sheet` (or `NavigationLink` push — planner discretion based on whether the existing HomeView NavigationStack supports the push without contortion) listing the 8 placeholder games with title + symbol. Tapping any of those still surfaces the existing `ComingSoonOverlay` pattern (1.8s auto-dismiss) — the sheet is a directory, not a launch surface.
- **D-05:** **`SHELL-05` requirement ID added to REQUIREMENTS.md.** Phase 6.1, satisfied by this plan. Description: "Home shows 2 game-tile cards in a 2-column square grid: Mines (enabled) + Upcoming (opens list of planned games)."

### Issue 2 — Flag-mode toggle for Minesweeper (MINES-12)

- **D-06:** **Mode swaps tap action; long-press is the OPPOSITE of current mode.** Two interaction modes: `.reveal` (tap = reveal / long-press = flag — current MINES-02 contract) and `.flag` (tap = toggle flag / long-press = reveal). Long-press always provides quick-action escape from the current mode. The 0.25s long-press threshold (ROADMAP SC1 lock) is preserved unchanged — only the *meaning* of the gesture flips per mode.
- **D-07:** **Default mode = `.reveal`.** Match the existing MINES-02 contract. New games start in reveal mode regardless of the user's last-used mode in the previous game.
- **D-08:** **Mode resets per-game.** Every `restart()` returns the VM to `.reveal` mode. No persistence across games (no UserDefaults flag for last-used mode). Simplest mental model — the mode is a transient mid-game ergonomic, not a persistent preference. (User declined the UserDefaults persistence option explicitly.)
- **D-09:** **Mode toggle UI = floating button below the board (FAB-style).** Lives BELOW `MinesweeperBoardView`, ABOVE the safe-area inset, in `MinesweeperGameView.body`. Visual: circular DKButton-shape, ~64pt diameter (≥44pt HIG min), tinted by `theme.colors.accentPrimary` in `.reveal` mode and `theme.colors.danger` (or a flag-yellow if added later) in `.flag` mode. Icon swaps: `cursorarrow.click` (reveal) ↔ `flag.fill` (flag). Tap = toggle mode. Light haptic on toggle (reuse `.sensoryFeedback(.impact(.light))`).
- **D-10:** **`MINES-12` requirement ID added to REQUIREMENTS.md.** Phase 6.1, satisfied by this plan. Description: "Minesweeper supports a Reveal/Flag interaction mode toggle. Tap action depends on current mode; long-press always performs the opposite action. Default = Reveal; resets per-game."
- **D-11:** **VM contract additions (Foundation-only, preserved):**
  - New `enum MinesweeperInteractionMode: String, Sendable, Equatable, Codable { case reveal, flag }`.
  - VM gains `private(set) var interactionMode: MinesweeperInteractionMode = .reveal`.
  - VM gains `func toggleInteractionMode()` — flips between cases, bumps a `modeToggleCount: Int` for haptic gating.
  - VM `restart()` resets `interactionMode = .reveal`.
  - **Tap routing inside the VM (NOT the view):** introduce `func handleTap(at: MinesweeperIndex)` and `func handleLongPress(at: MinesweeperIndex)` on the VM. Each branches on `interactionMode` and calls the existing `reveal(at:)` or `toggleFlag(at:)`. Cell view stops calling `reveal` / `toggleFlag` directly — it always calls `handleTap` / `handleLongPress`. This keeps mode-routing logic OUT of the view tier (CLAUDE.md §1 lightweight MVVM + ARCHITECTURE Anti-Pattern 1).
- **D-12:** **Existing P3/P5 invariants preserved:** the `.exclusively(before:)` gesture composition stays; the 0.25s `LongPressGesture` minimum stays; the `.sensoryFeedback(.selection)` / `.sensoryFeedback(.impact(.light))` cell-level haptic gating stays exactly as Plan 05-06 shipped them. Mode toggle adds a NEW haptic surface on the FAB but does NOT alter the cell-level haptic contract.

### Issue 3 — Hard board horizontal scroll fix + pinch-zoom graduation (A11Y-V2-02 → v1)

- **D-13:** **Auto-scale `cellSize` to available width.** Replace the fixed `cellSize` switch in `MinesweeperBoardView` (lines 45-51) with a `GeometryReader`-based formula:
  ```swift
  cellSize = max(
      Self.minCellSize,                                    // floor: 18pt (Discretion #1)
      (availableWidth - 2 * theme.spacing.l - CGFloat(cols - 1) * theme.spacing.xs) / CGFloat(cols)
  )
  ```
  Easy 9 cols on 390pt iPhone → ~38pt cells (similar to current 44pt). Hard 30 cols on 390pt iPhone → ~10pt cells, clamped UP to 18pt floor. Cells below floor → horizontal scroll re-engages BUT pinch-zoom (D-14) gives the user the escape hatch.
- **D-14:** **Graduate `A11Y-V2-02` (pinch-zoom on Hard board) to v1.0.** Pinch gesture wraps the board view. Scale range **0.8× – 2.0×** (Claude's Discretion #4). Below 1.0× = "see the whole board" (zoomed out so the smallest screens fit Hard); above 1.0× = "tap accuracy" (cells expand to comfortable HIG-min). Pinch state persists across `restart()` within a session (resets on cold launch). Pinch is available on ALL difficulties, not just Hard — but the floor and the auto-scale make Easy / Medium pinch-able for accessibility, not necessity.
- **D-15:** **REQUIREMENTS.md update:** flip `A11Y-V2-02: Pinch-to-zoom on Hard board` from v2 deferred → v1 row `A11Y-05: Pinch-to-zoom on Mines board (any difficulty); scale range 0.8x–2.0x; persists across restart within session`. Update Coverage line from `35` → `36 ✓`.
- **D-16:** **Auto-scale eliminates the horizontal `ScrollView` for ≥iPhone 13 widths.** `MinesweeperBoardView`'s `scrollAxis(for:)` helper either returns `[]` always (auto-scale handles it) OR keeps `.horizontal` as a fallback for sub-floor cases (planner picks based on whether iPhone SE Hard@floor still overflows — math says `30 × 18pt + spacing ≈ 600pt > 320pt`, so SE WILL still overflow even at floor; `.horizontal` fallback STAYS for that case). The user's "ugly horizontal scroll" complaint is resolved by:
  1. Cells auto-fit on standard phone widths (390pt+).
  2. iPhone SE users get pinch-zoom-out + the existing horizontal scroll as a fallback.

### Claude's Discretion

The user did not lock the following — planner has flexibility but should align with research / CLAUDE.md / ARCHITECTURE.md:

1. **Min cell size floor** (D-13) — recommend **18pt**. Tap-target tolerable for casual play; pinch-zoom (D-14) is the accessibility escape hatch. Alternative: 22pt (more conservative tap accuracy, but Hard cells stay visible scroll-territory longer on smaller phones).
2. **Upcoming tile presentation** (D-04) — recommend `.sheet` modal (medium detent on iOS 16+, full on smaller phones) over `NavigationLink` push. Sheet preserves the Home NavigationStack for Mines and reads as "directory of future games" rather than "go somewhere new." Planner picks.
3. **FAB icon set** (D-09) — recommend SF Symbols `cursorarrow.click` (reveal mode) ↔ `flag.fill` (flag mode). Alternatives: `hand.tap` (reveal) ↔ `flag` (flag), or `circle.dashed` ↔ `flag.fill`. Pick whichever reads cleanly at 28pt symbol size.
4. **Pinch-zoom scale range** (D-14) — recommend **0.8× – 2.0×**. Below 0.8× the cells become illegible; above 2.0× the user might as well play on a different difficulty. If user testing surfaces a different sweet spot, adjust in v1.0.x.
5. **Pinch-zoom centerpoint** (D-14) — recommend pivot on the gesture centroid (default `MagnificationGesture` behavior). Two-finger drag-to-pan IS expected during zoom — investigate whether SwiftUI's nested `MagnificationGesture` + `DragGesture` composes cleanly with the existing `LongPressGesture(0.25).exclusively(before: TapGesture())` cell-level gesture, OR whether we need a unified `SimultaneousGesture` umbrella at the BoardView level. Researcher should answer before planner picks.
6. **Mode-toggle FAB color** (D-09) — recommend `theme.colors.accentPrimary` for reveal and `theme.colors.danger` for flag. Alternative: a fixed flag-yellow (would need a new DesignKit token; D-19 says no new tokens for v1.0.x → stick with `danger`). The `danger` use here is non-error semantic (flag mode ≠ failure) — comment in source explaining the token choice.
7. **REQUIREMENTS.md edit timing** (D-10 + D-15) — recommend the edit lands in Plan 06.1-03 commit alongside the pinch-zoom code (the requirement and the implementation ship together). Alternative: a tail "docs(06.1-04): refresh REQUIREMENTS for SHELL-05 + MINES-12 + A11Y-05" commit if 06.1-01/02/03 each touch REQUIREMENTS for one ID and we want the doc-drift visible in a single commit.
8. **`UpcomingGamesView` file location** (D-04) — recommend `Screens/UpcomingGamesView.swift` (sibling to HomeView). Alternative: keep inside HomeView as a private nested view if it stays under ~50 lines (CLAUDE.md §8.1 file-size guidance).
9. **Pinch-zoom session-persistence boundary** (D-14) — recommend in-memory only (`@State` on the BoardView or VM). If user wants persistence across cold launches, that's a v1.0.x follow-up.
10. **Whether to bump the file-size soft cap on `MinesweeperGameView.swift`** — current 242 lines + FAB + mode wiring may push past CLAUDE.md §8.1 ~400-line cap. If so, extract `MinesweeperModeToggleFAB.swift` as a sibling (P5 05-04 AcknowledgmentsView precedent — same pattern, same justification).

## Specific Ideas

### Issue 1 — Home cards
- **Layout:** `LazyVGrid(columns: [GridItem(.flexible(), spacing: theme.spacing.m), GridItem(.flexible(), spacing: theme.spacing.m)], spacing: theme.spacing.m)` — 2 flexible columns, equal-width.
- **Tile aspect:** `.aspectRatio(1, contentMode: .fit)` on each DKCard — square at every screen width.
- **Mines tile content:** large icon (`square.grid.4x3.fill`, `.font(.system(size: 56))`) + title `Minesweeper` (`.headline`) + small caption "Tap to play".
- **Upcoming tile content:** large icon (`sparkles` or `gamecontroller`, `.font(.system(size: 56))`) + title `Upcoming` (`.headline`) + small caption "8 games coming".
- **Upcoming list (sheet):** a `List` of 8 rows, each with the existing per-game symbol + title. Tap any row → `ComingSoonOverlay` 1.8s dismiss. Sheet has a "Done" button in the nav bar.
- **REQUIREMENT:** `SHELL-05: Home shows 2 game-tile cards in a 2-column square grid: Mines (enabled) + Upcoming (opens list of planned games).`

### Issue 2 — Flag-mode toggle
- **VM addition:** `enum MinesweeperInteractionMode: String, Sendable, Equatable, Codable { case reveal, flag }` (Foundation-only).
- **VM API:** `private(set) var interactionMode: MinesweeperInteractionMode = .reveal`, `private(set) var modeToggleCount: Int = 0`, `func toggleInteractionMode()`, `func handleTap(at: MinesweeperIndex)`, `func handleLongPress(at: MinesweeperIndex)`. `restart()` resets `interactionMode = .reveal`.
- **CellView gesture handler change:** existing `onTap(index)` now calls `vm.handleTap(at: index)`; existing `onLongPress(index)` calls `vm.handleLongPress(at: index)`. The view does not branch on mode itself.
- **GameView FAB:** circular button below `MinesweeperBoardView`, ~64pt diameter, icon swap (`cursorarrow.click` ↔ `flag.fill`), color tint (`theme.colors.accentPrimary` ↔ `theme.colors.danger`), `.sensoryFeedback(.impact(.light), trigger: hapticsEnabled ? modeToggleCount : 0)`.
- **REQUIREMENT:** `MINES-12: Minesweeper supports a Reveal/Flag interaction mode toggle. Tap action depends on current mode; long-press always performs the opposite. Default = Reveal; resets per-game.`

### Issue 3 — Auto-scale + pinch-zoom
- **Auto-scale formula:** `cellSize = max(18, (availableWidth - 2 * theme.spacing.l - CGFloat(cols - 1) * theme.spacing.xs) / CGFloat(cols))`. `availableWidth` read via `GeometryReader` wrapping the LazyVGrid.
- **Pinch-zoom layer:** `.scaleEffect(zoomScale, anchor: .center)` + `.gesture(MagnificationGesture().onChanged { ... }.onEnded { ... })`. State: `@State private var zoomScale: CGFloat = 1.0`, clamped to `[0.8, 2.0]` on every gesture update.
- **Gesture composition validation (Discretion #5):** confirm during research that `MagnificationGesture` at BoardView level + `LongPressGesture.exclusively(before: TapGesture())` at CellView level compose without misfire. If conflict surfaces, wrap CellView gesture in `.simultaneously(with: TapGesture())` umbrella OR scope MagnificationGesture to a dedicated zoom-mode toggle.
- **REQUIREMENT:** `A11Y-05: Pinch-to-zoom on Mines board (any difficulty); scale range 0.8x–2.0x; persists across restart within session.` (Graduates A11Y-V2-02 from v2 → v1.)
