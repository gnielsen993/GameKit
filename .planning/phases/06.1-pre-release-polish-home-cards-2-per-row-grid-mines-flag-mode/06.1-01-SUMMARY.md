### xcstringstool sync command flag

The plan §Action block suggested:
```
xcrun --sdk iphoneos xcstringstool sync \
    --positional-arguments \
    gamekit/Resources/Localizable.xcstrings \
    $(find gamekit -name "*.swift" -not -path "*/Build/*")
```

The actual `xcstringstool sync` invocation requires `--stringsdata <stringsdata>...` (NOT `--positional-arguments` with .swift files). The .stringsdata files are generated by xcodebuild during compilation and live under DerivedData. The corrected invocation:

```bash
find /Users/.../DerivedData/.../Build/Intermediates.noindex \
  -path '*gamekit.build/Debug-iphonesimulator*Objects-normal/arm64*' \
  -name "*.stringsdata" -print0 \
| xargs -0 xcrun xcstringstool sync gamekit/Resources/Localizable.xcstrings --stringsdata
```

The `-print0 | xargs -0` form is load-bearing — paths in DerivedData contain no spaces but the shell's word-splitting of `$(find ...)` was being interpreted as a single argument by xcstringstool, which produced the misleading "Couldn't read stringsdata file at '<concatenated paths>'" warning before silently no-op'ing. After fixing to null-separated input, all 3 new keys appeared in the catalog correctly. Pattern locked for future xcstringstool sync invocations across the project.
