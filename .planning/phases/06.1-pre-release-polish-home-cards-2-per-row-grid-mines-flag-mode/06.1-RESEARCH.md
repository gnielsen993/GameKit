## Summary

Phase 6.1 ships three focused UI changes that are independently implementable but share a codebase. The core research questions are answered with HIGH confidence from SDK interface analysis and verified iOS 17+ SwiftUI documentation.

**Load-bearing finding (gesture composition):** `MagnifyGesture` (iOS 17+, not the deprecated `MagnificationGesture`) applied via `.simultaneousGesture(MagnifyGesture()...)` on `MinesweeperBoardView`'s container COEXISTS CLEANLY with child cell `LongPressGesture(0.25).exclusively(before: TapGesture())`. The `.simultaneousGesture` modifier does not compete with child gestures — it runs alongside them. Single-finger taps/long-presses hit the child gesture (which wins by default child-priority). Two-finger pinch hits the parent simultaneously. No `.simultaneously(with:)` umbrella is needed at the child level; the existing `.exclusively(before:)` composition is preserved verbatim per D-12.

**Load-bearing finding (auto-scale sizing):** The safest pattern for reading available width in iOS 17+ is `Color.clear.frame(maxWidth: .infinity).onGeometryChange(for: CGFloat.self, of: { $0.size.width }, action: { availableWidth = $0 })` as a background/overlay on the container, avoiding GeometryReader's greedy layout expansion inside ScrollView. `onGeometryChange` is available iOS 16+ (confirmed from SDK interface). GeometryReader placed directly INSIDE a ScrollView is the primary layout-loop trigger — the background trick avoids it.

**Primary recommendation:** For all three issues, use the smallest-change patterns already proven in this codebase. No new architectural layers required.

### Pattern 1: 2-col LazyVGrid with Square Cards

**What:** Two `GridItem(.flexible())` columns. Each card uses `.aspectRatio(1, contentMode: .fit)` to enforce square shape at any screen width.
**When to use:** Home screen with a fixed small number of tiles (2 in this phase).

```swift
// Source: CONTEXT.md §Specific Ideas Issue 1 + LazyVGrid iOS 14+ standard
LazyVGrid(
    columns: [
        GridItem(.flexible(), spacing: theme.spacing.m),
        GridItem(.flexible(), spacing: theme.spacing.m)
    ],
    spacing: theme.spacing.m
) {
    ForEach(homeTiles) { tile in
        TileView(tile: tile, theme: theme)
            .aspectRatio(1, contentMode: .fit)
    }
}
.padding(.horizontal, theme.spacing.m)
```

**Note:** `GridItem(.flexible())` divides available width equally. At 390pt screen with `theme.spacing.m` (12pt) gap and `theme.spacing.m` (12pt) horizontal padding each side: each tile ≈ (390 - 24 - 12) / 2 ≈ 177pt square. Plenty of room for icon + title + caption.

### Pattern 3: MagnifyGesture via .simultaneousGesture on BoardView Container

**What:** Apply pinch-zoom at the container (BoardView) level using `.simultaneousGesture(MagnifyGesture()...)`. This is the key pattern for coexisting with child cell gestures.
**When to use:** Whenever a parent container needs a multi-finger gesture while child views own single-finger gestures.

```swift
// Source: Apple SwiftUI gesture priority rules + CONTEXT.md Discretion #5
// MagnifyGesture confirmed @available(iOS 17.0, ...) from SDK swiftinterface [VERIFIED]
// .simultaneousGesture does NOT block child gesture recognition [VERIFIED: dev.to gesture internals]

struct MinesweeperBoardView: View {
    // ... existing props ...
    @State private var zoomScale: CGFloat = 1.0
    @State private var baseZoomScale: CGFloat = 1.0   // accumulated across gestures

    var body: some View {
        ScrollView(scrollAxis(for: board.difficulty), showsIndicators: false) {
            LazyVGrid(columns: columns, spacing: theme.spacing.xs) {
                ForEach(board.allIndices(), id: \.self) { index in
                    // ... existing MinesweeperCellView with unchanged gesture ...
                }
            }
            .padding(.horizontal, theme.spacing.l)
            .padding(.vertical, theme.spacing.s)
        }
        .scaleEffect(zoomScale, anchor: .center)
        .simultaneousGesture(
            MagnifyGesture()
                .onChanged { value in
                    // Clamp during gesture: base * delta, clamped to [0.8, 2.0]
                    zoomScale = min(2.0, max(0.8, baseZoomScale * value.magnification))
                }
                .onEnded { value in
                    baseZoomScale = min(2.0, max(0.8, baseZoomScale * value.magnification))
                    zoomScale = baseZoomScale
                }
        )
    }
}
```

**Why dual @State (zoomScale + baseZoomScale):** `MagnifyGesture.Value.magnification` is the delta multiplier from the START of the current gesture (e.g., 1.0 = no change, 1.5 = 50% bigger), not an absolute scale. Applying it directly would reset zoom on each new pinch gesture. Two-state pattern (base × delta, commit to base on end) preserves accumulated zoom across gestures. Pattern verified from serialcoder.dev MagnifyGesture article. [MEDIUM: CITED: serialcoder.dev/magnifygesture]

**Why .simultaneousGesture and not .gesture:** `.gesture(MagnifyGesture())` on the parent creates a competing gesture. SwiftUI gives child gestures default priority. With `.gesture(...)` on parent and `.exclusively(before:)` on child, the child exclusively wins single-finger interactions AND a two-finger pinch on a cell would still be consumed by the child's `.exclusively(before:)` gesture recognizer. `.simultaneousGesture(...)` on the parent marks the parent gesture as non-competing — both can recognize independently. The child's `.exclusively(before:)` only matters for LongPress vs Tap disambiguation; it does not affect multi-finger recognition. [MEDIUM: dev.to gesture internals article + SwiftUI simultaneous gesture guide]

### Pattern 4: Auto-Scale cellSize via onGeometryChange Background

**What:** Measure available width without using GeometryReader as a layout participant. Use `.onGeometryChange(...)` as a background modifier to capture width once and update `@State var availableWidth`.
**When to use:** Any time you need to read a dimension to compute layout parameters (like cellSize) and want to avoid layout loops.

```swift
// Source: fatbobman.com geometry snippet + Apple SDK onGeometryChange available iOS 16+ [VERIFIED]
// Applied to MinesweeperBoardView

struct MinesweeperBoardView: View {
    @State private var availableWidth: CGFloat = 390  // sensible default for 1st render
    // ... other props ...

    private var cellSize: CGFloat {
        max(
            Self.minCellSize,   // 18pt floor (Claude's Discretion #1, locked by D-13)
            (availableWidth - 2 * theme.spacing.l - CGFloat(board.cols - 1) * theme.spacing.xs)
            / CGFloat(board.cols)
        )
    }

    private static let minCellSize: CGFloat = 18  // intrinsic component dimension (P3 D-12 precedent)

    var body: some View {
        ScrollView(scrollAxis(for: board.difficulty), showsIndicators: false) {
            LazyVGrid(columns: columns, spacing: theme.spacing.xs) {
                // ... cells ...
            }
            .padding(.horizontal, theme.spacing.l)
            .padding(.vertical, theme.spacing.s)
        }
        .background(
            Color.clear
                .onGeometryChange(for: CGFloat.self, of: { $0.size.width }) { width in
                    availableWidth = width
                }
        )
        // pinch-zoom applied above (Pattern 3)
    }
}
```

**Why background, not direct:** `onGeometryChange` as a background modifier runs in the background layout pass and does not expand the view. If placed directly on the ScrollView or LazyVGrid, the closure runs on the same layout pass that uses `availableWidth`, creating a potential two-pass settle. The background trick breaks the cycle: background reads width after parent layout, updates @State, causing a single follow-up render. Apple's own `swiftwithmajid.com` article explicitly recommends this background/overlay pattern to prevent GeometryReader expansion. [MEDIUM: CITED: swiftwithmajid.com/geometryreader + fatbobman.com geometry]

**Width formula sanity check (verified arithmetic):**
- iPhone 16 Pro (393pt logical): `(393 - 32 - 29*4) / 30 = (393-32-116)/30 = 245/30 ≈ 8.2pt` → clamped to 18pt floor → Hard board still slightly wider than screen → horizontal scroll re-engages (D-16 expected behavior)
- iPhone 15 (390pt): `(390 - 32 - 116)/30 = 242/30 ≈ 8.1pt` → floor 18pt → same
- At 18pt floor: `30 × 18 + 31 × 4 = 540 + 124 = 664pt` vs screen 390pt → scroll needed. This is exactly D-16 behavior: floor cells + pinch-zoom-out gives the user the exit.
- Easy at 390pt: `(390-32-8*4)/9 = (390-32-32)/9 = 326/9 ≈ 36.2pt` (slightly below current 44pt but fine; pinch-zoom available for accessibility if needed)
- Medium at 390pt: `(390-32-15*4)/16 = (390-32-60)/16 = 298/16 ≈ 18.6pt` → clamped to 18pt floor → Medium now needs horizontal scroll on standard iPhone. **This is a regression risk.** Planner should note: consider `theme.spacing.xs / 2` cell spacing for medium/easy, OR use a minimum padding of `theme.spacing.s` instead of `theme.spacing.l` on each side for Hard only, OR accept the floor and rely on pinch-zoom.

**PLANNING ALERT:** The formula as written in CONTEXT D-13 uses `2 * theme.spacing.l` (32pt) horizontal padding. At theme.spacing.xs (4pt) cell gap, Medium 16-col will hit the floor on 390pt width. This may be unintentional — Easy/Medium worked fine without scroll before. Recommend planner verify: for Easy/Medium, the formula should produce cells ≥ 30pt without hitting the floor. Either (a) use `theme.spacing.s` (8pt) instead of `theme.spacing.l` as padding for the grid, keeping `theme.spacing.l` only as visual padding, or (b) apply separate min floors per difficulty (Easy: 36pt, Medium: 30pt, Hard: 18pt). The auto-scale intent is primarily to fix HARD; it must not degrade Easy/Medium.

### Pattern 5: VM Interaction-Mode Routing (Foundation-only)

**What:** Enum + two dispatch methods on MinesweeperViewModel. The view always calls `handleTap(at:)` / `handleLongPress(at:)`. The VM branches on `interactionMode` to call the correct underlying action. This keeps mode logic out of the view tier (CLAUDE.md §1 + D-11).

```swift
// Foundation-only — no import SwiftUI needed. Add to MinesweeperViewModel.swift.
// Source: CONTEXT.md D-11 + CLAUDE.md §4 (smallest change)

enum MinesweeperInteractionMode: String, Sendable, Equatable, Codable {
    case reveal
    case flag
}

// Inside MinesweeperViewModel:
private(set) var interactionMode: MinesweeperInteractionMode = .reveal
private(set) var modeToggleCount: Int = 0

func toggleInteractionMode() {
    interactionMode = (interactionMode == .reveal) ? .flag : .reveal
    modeToggleCount += 1
}

func handleTap(at index: MinesweeperIndex) {
    switch interactionMode {
    case .reveal: reveal(at: index)
    case .flag:   toggleFlag(at: index)
    }
}

func handleLongPress(at index: MinesweeperIndex) {
    switch interactionMode {
    case .reveal: toggleFlag(at: index)   // opposite of reveal mode
    case .flag:   reveal(at: index)       // opposite of flag mode
    }
}

// In restart():
interactionMode = .reveal   // D-08
// modeToggleCount intentionally NOT reset — it's only a haptic trigger counter
// and a fresh game that starts in .reveal has no toggle events yet; leaving it
// non-zero does not affect behavior since the sensoryFeedback trigger compares
// value-change, not zero-crossing. Optionally reset to 0 for cleanliness.
```

**Why Codable on MinesweeperInteractionMode:** The enum is Codable in case a future plan wants to persist last-used mode (D-08 deferred item). Codable has zero runtime cost if unused. [ASSUMED: future-proofing reasoning]

```swift
            .frame(width: 64, height: 64)   // intrinsic component dimension per D-19
            .background(fabTintColor)
            .clipShape(Circle())

    .buttonStyle(.plain)
    .contentShape(Circle())
    .accessibilityLabel(viewModel.interactionMode == .reveal

                        : Text("Switch to Reveal mode"))

                     trigger: settingsStore.hapticsEnabled ? viewModel.modeToggleCount : 0)

    case .reveal: return theme.colors.accentPrimary
    case .flag:   return theme.colors.danger

    }

```

**64pt sizing rationale:** HIG minimum touch target is 44pt. 64pt exceeds minimum and follows Material Design FAB convention (56-64pt standard). It is an intrinsic component dimension per D-19 precedent (same as the locked cell sizes in P3 D-12). The literal `64` lives in the FAB view frame only, not as a spacing token. ### Anti-Patterns to Avoid

- **GeometryReader as a first child of ScrollView or LazyVGrid:** GeometryReader greedily fills all available space and causes height calculation cycles. Use `.onGeometryChange` as a `.background()` on the container instead.
- **MagnificationGesture (old API):** Produces deprecation warnings. The iOS SDK explicitly marks it `deprecated: 100000.0, renamed: "MagnifyGesture"`. Use `MagnifyGesture` (iOS 17+).
- **mode routing in the cell view:** CellView must NOT branch on `interactionMode` to decide what to call. It calls `vm.handleTap(at:)` and `vm.handleLongPress(at:)` and lets the VM dispatch. This is the ARCHITECTURE Anti-Pattern 1 lock from CONTEXT D-11.
- **Passing interactionMode through 3 layers of view props:** BoardView does NOT need `interactionMode` as a prop — it only passes `onTap: vm.handleTap` and `onLongPress: vm.handleLongPress` closures. The mode is invisible to the view hierarchy except for the FAB tint (which reads `vm.interactionMode` directly at GameView level).
- **Using .gesture(MagnifyGesture()) instead of .simultaneousGesture(MagnifyGesture()):** `.gesture(...)` on the parent competes with child gestures. `.simultaneousGesture(...)` allows both parent and child to recognize without conflict.
- **Resetting baseZoomScale in restart():** D-14 says zoom persists across restart within session. Do not reset zoom state in `restart()`. Since zoom state lives in the View (@State), this is naturally honored — the VM's `restart()` cannot reach view state.

## Common Pitfalls

### Pitfall 1: Using MagnificationGesture Instead of MagnifyGesture
**What goes wrong:** Deprecation warnings flood the build log. The old API's zoom origin was fixed to the top-left on iOS 16 pinch-in; MagnifyGesture fixes this with `startAnchor`.
**Why it happens:** Training data and many online tutorials still show `MagnificationGesture`. Context7 and the iOS SDK are the authoritative sources.
**How to avoid:** Use `MagnifyGesture` (iOS 17+ confirmed `@available`). The `.simultaneousGesture(MagnifyGesture()...)` idiom is the correct form.
**Warning signs:** Build warning `'MagnificationGesture' is deprecated: renamed to 'MagnifyGesture'`.

### Pitfall 2: GeometryReader Inside ScrollView Causes Layout Loop
**What goes wrong:** The ScrollView offers infinite height. GeometryReader fills all available space. Its content then lays out at infinite height, causing the board to render as a single line or with bizarre spacing.
**Why it happens:** GeometryReader's default behavior is to fill all proposed space, which is unbounded inside a ScrollView.
**How to avoid:** Use `.onGeometryChange(for: CGFloat.self, of: { $0.size.width }) { w in availableWidth = w }` as a `.background(Color.clear...)` on the outer ScrollView. The background constraint prevents the measurement view from affecting layout.
**Warning signs:** Board renders with zero height, extreme height, or all cells on a single row.

### Pitfall 3: Medium/Easy Board Hits cellSize Floor After Auto-Scale Formula
**What goes wrong:** The D-13 formula with `2 * theme.spacing.l` horizontal padding produces cells of ~18pt for MEDIUM (16-col) on standard iPhone widths, enabling horizontal scroll where none existed before.
**Why it happens:** The formula was designed for the Hard (30-col) regression problem; it undersizes Easy/Medium when applied uniformly with large padding.
**How to avoid:** Either (a) use `theme.spacing.s` or `theme.spacing.xs` as horizontal padding for the cell grid itself (keep `theme.spacing.l` for visual breathing room via a different container), OR (b) set per-difficulty floors: Easy ≥ 36pt, Medium ≥ 30pt, Hard ≥ 18pt. Planner must decide and verify Math before shipping.
**Warning signs:** Easy/Medium board gains a horizontal scroll bar that did not exist before P6.1.

### Pitfall 4: CellView Gesture Misfires When onTap/onLongPress Closure Target Changes
**What goes wrong:** If `vm.handleTap` captures a stale closure reference (e.g., if it was captured in init instead of being computed at call time), mode changes don't take effect until the next render.
**Why it happens:** In SwiftUI, closures passed as props to child views are captured at render time. If the closure itself changes meaning without the prop reference changing, SwiftUI may not redraw.
**How to avoid:** Pass closures that read VM state at call time: `onTap: { vm.handleTap(at: $0) }` (inline closure captures vm by reference from @State). Do NOT pass `onTap: vm.handleTap` directly as a function reference — that would capture a snapshot.
**Warning signs:** Mode toggle visually updates the FAB but cells still behave in the old mode. Fix: always use `{ vm.handleTap(at: $0) }` wrapper form.

### Pitfall 5: .simultaneousGesture Still Interfering with Cell Long-Press
**What goes wrong:** On very slow long-press (>1s), some devices report the MagnifyGesture recognizer interpreting two close-together successive taps as a magnification gesture, causing the cell long-press to be eaten.
**Why it happens:** If `minimumScaleDelta` threshold is too low, `MagnifyGesture` activates on multi-touch noise.
**How to avoid:** Use `MagnifyGesture(minimumScaleDelta: 0.05)` (slightly above default `0.01`) to reduce false activations on close two-finger contact. The 50-tap SC1 manual test (03-04-PLAN recipe) must be run after implementing pinch-zoom to confirm zero misfires.
**Warning signs:** Long-press to flag occasionally fails; SC1 manual test shows ≥1 misfire in 50 taps.

### Pitfall 6: vm.handleTap / vm.handleLongPress Missing the vmSourceFile_importsOnlyFoundation Grep Gate
**What goes wrong:** If `MinesweeperInteractionMode` is declared in a new file that accidentally imports SwiftUI, the structural test `vmSourceFile_importsOnlyFoundation` fails.
**Why it happens:** Adding a new file to the `Games/Minesweeper/` folder that is not Foundation-only.
**How to avoid:** Declare `MinesweeperInteractionMode` inside `MinesweeperViewModel.swift` (file-scope, before the class). The structural grep test targets the VM source file specifically, so a sibling file with a SwiftUI import does not break the VM purity invariant directly — but CLAUDE.md §1 still forbids it in `Games/`. If a sibling enum file is needed, it must be Foundation-only.
**Warning signs:** `vmSourceFile_importsOnlyFoundation` structural test fails. Build warning about unused SwiftUI import.

### Pitfall 7: Upcoming Sheet NavigationStack Nesting
**What goes wrong:** Presenting UpcomingGamesView inside a `.sheet` without wrapping in a `NavigationStack` means there's no `.navigationTitle`, no toolbar for "Done" button, and the sheet has no affordance for programmatic dismiss.
**Why it happens:** Sheets do not inherit the presenter's NavigationStack.
**How to avoid:** Wrap UpcomingGamesView in a `NavigationStack` INSIDE the sheet closure. Do NOT let HomeView's NavigationStack extend into the sheet. This follows the same pattern as every other sheet in the codebase (e.g., the stats export fileExporter). The sheet gets its own navigation context.
**Warning signs:** "Done" button missing; title missing; sheet cannot be dismissed except by drag.

### Pitfall 8: Pre-commit Hook Firing on FAB Frame Literal
**What goes wrong:** The FOUND-07 pre-commit hook rejects `padding(<int>)` and `cornerRadius: <int>`. The FAB uses `.frame(width: 64, height: 64)` which is NOT a spacing token and NOT a cornerRadius — but `padding(16)` inside the FAB would fire the hook.
**Why it happens:** The hook targets `padding\(<int>)` and `cornerRadius: <int>`. Frame width/height literals are explicitly exempted as "intrinsic component dimensions" per D-19 and the P3 D-12 precedent.
**How to avoid:** Use `theme.spacing.m` for `.padding(.trailing, ...)` and `.padding(.bottom, ...)` around the FAB. Use `theme.spacing.m` for internal icon padding inside the button label if any. The `64` in `.frame(width: 64, height: 64)` is a component dimension, not a spacing token — it does not trigger the hook. [VERIFIED: hook targets `padding\(` not `frame\(`]

```swift
// Applied to MinesweeperBoardView.swift (currently 116 lines; stays well under 500 cap)

@State private var availableWidth: CGFloat = 390
@State private var zoomScale: CGFloat = 1.0
@State private var baseZoomScale: CGFloat = 1.0

private static let minCellSize: CGFloat = 18  // D-13 + Discretion #1

private var cellSize: CGFloat {
    max(
        Self.minCellSize,
        (availableWidth
         - 2 * theme.spacing.l
         - CGFloat(board.cols - 1) * theme.spacing.xs)
        / CGFloat(board.cols)
    )
}

var body: some View {
    ScrollView(scrollAxis(for: board.difficulty), showsIndicators: false) {
        LazyVGrid(columns: columns, spacing: theme.spacing.xs) {
            // ...existing cells unchanged...
        }
        .padding(.horizontal, theme.spacing.l)
        .padding(.vertical, theme.spacing.s)
    }
    .background(
        Color.clear
            .onGeometryChange(for: CGFloat.self, of: { $0.size.width }) { w in
                availableWidth = w
            }
    )
    .scaleEffect(zoomScale, anchor: .center)
    .simultaneousGesture(
        MagnifyGesture(minimumScaleDelta: 0.05)
            .onChanged { value in
                zoomScale = min(2.0, max(0.8, baseZoomScale * value.magnification))
            }
            .onEnded { value in
                baseZoomScale = min(2.0, max(0.8, baseZoomScale * value.magnification))
                zoomScale = baseZoomScale
            }
    )
}
```

## Open Questions

1. **Medium/Easy cellSize floor arithmetic**
   - What we know: The D-13 formula with `theme.spacing.l` padding produces sub-18pt cells for Medium on standard iPhone widths.
   - What's unclear: Is the intent to auto-scale all difficulties or only hard? Should Easy/Medium have higher floors?
   - Recommendation: Planner should verify arithmetic and set per-difficulty floors or reduce horizontal padding for the grid content (keeping visual padding in the outer container). CONTEXT.md D-13 does not explicitly address Easy/Medium regression — treat this as a gap that needs a concrete number from the planner.

2. **Whether to reset modeToggleCount in restart()**
   - What we know: `revealCount` and `flagToggleCount` are reset to 0 in restart() (verified in MinesweeperViewModel line 272-273). `modeToggleCount` is a haptic trigger counter, similar in nature.
   - What's unclear: Does resetting to 0 cause the haptic to NOT fire on the first toggle after restart (because value didn't "change" if both before and after are 0)? In SwiftUI `.sensoryFeedback(trigger:)`, the feedback fires on VALUE CHANGE, not on non-zero value. So resetting to 0 and then calling toggleInteractionMode() bumps to 1 — that IS a change — feedback fires correctly.
   - Recommendation: Reset `modeToggleCount = 0` in restart() for symmetry with other trigger counters.

3. **Accessibility label for FAB when game is in terminal state**
   - What we know: When the game ends (won/lost), the end-state overlay covers the board but the FAB is still in the VStack below.
   - What's unclear: Should the FAB be hidden or disabled in terminal state? The VoiceOver user navigating to the FAB after a loss would hear "Switch to Flag mode" which is irrelevant.
   - Recommendation: Add `.opacity(viewModel.terminalOutcome != nil ? 0 : 1).allowsHitTesting(viewModel.terminalOutcome == nil)` on the FAB container. This matches the existing pattern where end-state overlay covers interactive content.
