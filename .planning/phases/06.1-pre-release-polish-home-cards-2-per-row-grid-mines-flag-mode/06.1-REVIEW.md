## Medium

### MR-01: BoardLayoutTests assert against the wrong padding token

**File:** `gamekit/gamekitTests/Games/Minesweeper/MinesweeperBoardLayoutTests.swift:46-103`
**Issue:** All four `cellSize_*` tests pass `padding: 16` to the static
helper. The actual production callsite uses `padding: theme.spacing.s`
(8pt) — see `MinesweeperBoardView.cellSize` instance computed property
(line 117-124). The tests therefore verify the formula against an input
that production never uses.

The static helper itself is pure, so the tests still pass for the values
they assert — but the test docstrings say things like "Easy 9-col @ 390pt
should be ~36.2pt" which is the value at padding=16, NOT the production
value (~38pt at padding=8). Anyone who reads the test to predict on-screen
behavior will be off by ~2pt across all difficulties.

More importantly: at production padding=8pt and spacing=4pt, Medium 16-col
on iPhone 13/14/15 (390pt) computes to `(390 - 16 - 60)/16 ≈ 19.6pt`
which is just above the 18pt floor. There is no test that verifies this
narrow margin holds; a future tweak to `theme.spacing.s` from 8pt to,
say, 10pt would silently push Medium below the floor and re-enable the
horizontal scroll bar Phase 6.1 was explicitly designed to fix
(RESEARCH Pitfall 3 — "Medium/Easy Board Hits cellSize Floor").

**Fix:** Update the four cellSize tests to use `padding: 8, spacing: 4`
(matching the production callsite values), and add at least one test
that asserts Medium @ 390pt > floor:

```swift
@Test
func cellSize_medium_16col_390pt_atProductionPadding_clearsFloor() {
    // Production callsite: padding = theme.spacing.s (8pt), spacing = theme.spacing.xs (4pt).
    // Regression guard: if theme.spacing.s widens, Medium will fall to floor.
    let result = MinesweeperBoardView.cellSize(
        forWidth: 390,
        cols: 16,
        padding: 8,      // matches MinesweeperBoardView.cellSize instance prop
        spacing: 4
    )
    #expect(result > 18.0,
            "Medium @ 390pt at production padding must clear the floor; got \(result)")
}
```

### MR-02: GameView VStack does not isolate the FAB from the win-sweep / loss-shake animation surface

**File:** `gamekit/gamekit/Games/Minesweeper/MinesweeperGameView.swift:69-149`
**Issue:** The new mode-toggle FAB (lines 112-148) sits inside the same
VStack as `MinesweeperBoardView`, AFTER the `.keyframeAnimator` modifier
on the BoardView (line 93-104). The keyframe animator is bound to
`viewModel.phase.isLossShake` and applies an `.offset(x: value)`
modifier ONLY to the BoardView (it's chained to the BoardView, not the
VStack), so the FAB itself does not shake — that part is correct.

However, the FAB does appear inside the same outer ZStack as the
`.phaseAnimator` win-sweep `Rectangle` (lines 157-168). The win-sweep
Rectangle ignores safe area and uses `.allowsHitTesting(false)`, so it
cannot intercept the FAB's tap target — but during the sweep wash the
FAB icon will be tinted by the success-color overlay alpha (max 0.25).
This is cosmetic, not functional, but reads slightly off because the
FAB is supposed to be hidden post-terminal (`.opacity(0)` on lines
136-137). The `.opacity` happens in the same render frame as the
phase change to `.winSweep`, so on a fast device the user sees the
FAB fade out cleanly. On a slow device (or under `accessibilityReduceMotion`
where the sweep is `[0.0]`) the order is fine.

**Fix:** Either accept this as a one-frame visual artifact (low risk —
end-state overlay covers the FAB area within ~16ms anyway), or hoist the
FAB above the ZStack content layering. The simplest improvement is to
add a brief comment near line 136 explicitly documenting the layering
assumption:

```swift
// FAB hides instantly post-terminal so the win-sweep Rectangle wash does
// not tint the icon during the .phaseAnimator pass. The .opacity(0) here
// fires in the same render frame as the .winSweep phase set, so the
// fade-out happens before the wash visibly hits the FAB area.
.opacity(viewModel.terminalOutcome == nil ? 1 : 0)
```

### MR-03: BoardView availableWidth defaults to 0 — first paint produces minCellSize floor for Easy/Medium too

**File:** `gamekit/gamekit/Games/Minesweeper/MinesweeperBoardView.swift:57`
**Issue:** `@State private var availableWidth: CGFloat = 0` means on the
very first render — before `.onGeometryChange` fires its callback —
`cellSize` evaluates to `max(18, (0 - 16 - …) / 9)` = `18` for ALL
difficulties. The first render thus shows a tiny board with all cells
at the 18pt floor, then re-renders ~one frame later when
`.onGeometryChange` resolves the real width.

For Easy and Medium this produces a brief "tiny → correct size" pop on
scene appearance. The cells are `LazyVGrid(columns: …)` so the layout
reflows automatically, but the user can perceive the snap on a fast
device. This is the exact behavior RESEARCH Pattern 4 warns about and
the reason the recommended default in similar patterns is something
like `390` (a sensible iPhone width) rather than `0`.

The CONTEXT D-13 formula and the threat-model entry T-06.1-03-03
explicitly accept "no division-by-zero (cols >= 9 always); first-render
uses 18pt floor, second render after onGeometryChange uses computed
value (single follow-up render, not a loop)" — so this is documented
behavior. But the documentation is in the threat model, not in the
code, and the user-visible artifact (board pops from tiny to correct)
is more noticeable on Easy than the docs suggest.

**Fix:** Change the default to a sensible iPhone width so the first
render is already close to correct, with the `.onGeometryChange`
callback fine-tuning to the actual screen width:

```swift
// P6.1 (A11Y-05) — measured ScrollView width via .onGeometryChange.
// Default 390pt matches iPhone 13/14/15 logical width; the first render
// uses this estimate and .onGeometryChange refines on second render
// (single follow-up render, not a loop). This avoids the "all cells
// floor at 18pt then snap to correct size" first-paint pop that a
// 0-default would produce on every scene appearance.
@State private var availableWidth: CGFloat = 390
```

This is the value RESEARCH §Code Examples explicitly recommends ("State
private var availableWidth: CGFloat = 390 // sensible default for 1st
render"). The current `0` default appears to be a regression from the
research recommendation.

### LR-01: HomeView.tileCard takes Color directly — DesignKit token threading is one layer deep

**File:** `gamekit/gamekit/Screens/HomeView.swift:108-113`
**Issue:** The `tileCard(symbol:iconColor:title:caption:)` helper accepts
a `Color` parameter directly. Today it's only called from `minesTile`
and `upcomingTile` which pass `theme.colors.accentPrimary` and
`theme.colors.accentSecondary` respectively, so all values are tokens.
But the type signature does not enforce this — a future caller could
pass `Color.red` and the compiler would not complain. This is a stylistic
item, not a violation; the FOUND-07 hook only fires on `Color(red:`,
`Color("#…")`, etc., not on bare `.red`.

**Fix:** Either accept the looser type (it's a private helper with two
trusted callers, so the risk is low), OR tighten by passing a closure
that reads from the theme:

```swift
// Tighter shape — caller cannot pass a non-token color.
@ViewBuilder
private func tileCard(
    symbol: String,
    iconColor: KeyPath<Theme.Colors, Color>,
    title: String,
    caption: String
) -> some View {
    DKCard(theme: theme) {
        VStack(spacing: theme.spacing.s) {
            Image(systemName: symbol)
                .font(.system(size: 56))
                .foregroundStyle(theme.colors[keyPath: iconColor])
            // ...
        }
    }
}
```

Low risk — leaving as-is is fine. Calling out for future readers.

### LR-02: GameView FAB icon font size 28 + frame 64 are inline magic numbers

**File:** `gamekit/gamekit/Games/Minesweeper/MinesweeperGameView.swift:118-120`
**Issue:** `.font(.system(size: 28, weight: .semibold))` and
`.frame(width: 64, height: 64)` use bare numeric literals. Per
CONTEXT D-19 + RESEARCH §Pitfall 8 these are intrinsic component
dimensions (not spacing tokens) and are explicitly exempt from the
FOUND-07 hook — but the rationale is documented only in the planning
docs, not in the source comment alongside the literal.

**Fix:** Add a one-line comment near the literal that names the
exemption, so a future reader of the code (without the planning docs
in hand) doesn't mistake them for token violations:

```swift
Image(systemName: viewModel.interactionMode == .reveal
                  ? "cursorarrow.click"
                  : "flag.fill")
    // FAB icon size + 64pt diameter are intrinsic component dimensions
    // per CONTEXT D-19 (FAB ≥ 44pt HIG min, 64pt Material FAB convention)
    // — exempt from FOUND-07 spacing-token rule.
    .font(.system(size: 28, weight: .semibold))
    .foregroundStyle(theme.colors.textPrimary)
    .frame(width: 64, height: 64)
```

### LR-03: UpcomingGamesView upcomingGames is a file-private `let`, not a `static` constant — re-evaluated per import

**File:** `gamekit/gamekit/Screens/UpcomingGamesView.swift:64-73`
**Issue:** `private let upcomingGames: [GameCard]` at file scope is

struct UpcomingGamesView: View {

### LR-04: HomeView sheet's Task.sleep id-guard mutates @State from a captured async closure

**File:** `gamekit/gamekit/Screens/HomeView.swift:53-63`
**Issue:** The `.sheet` content closure captures `showingComingSoon`
(an `@State`) and writes to it both immediately (`showingComingSoon =
card`) and inside a `Task { @MainActor in … }` after a 1.8s sleep. The
id-guard `if showingComingSoon?.id == card.id` correctly prevents the
auto-dismiss from clearing a NEWER coming-soon toast, so this is sound.

The two minor concerns:
1. If the user taps the same Upcoming row twice in 1.8s, the second
   tap's Task will see `showingComingSoon?.id == card.id` (because
   they ARE the same id) and clear the toast at the second Task's
   1.8s mark. Net effect is "toast displays for at least 1.8s but
   may extend on rapid re-taps" — correct per the spec, but worth
   noting.
2. There is no test that locks this id-guard behavior. If the closure
   is ever simplified to `showingComingSoon = nil` without the guard,
   the rapid-retap race re-emerges. This is a nice candidate for a
   small UI/integration test if/when the project decides UI tests
   are worth the cost (CLAUDE.md §5 says minimal unless requested).

**Fix:** No code change required. Optional: add a one-line comment
explaining the id-guard:

```swift
Task { @MainActor in
    try? await Task.sleep(nanoseconds: 1_800_000_000)
    // Id-guard: only clear the toast if a newer tap has not replaced it.
    // Without this, rapid re-tap on a different game would cancel the
    // newer toast at the older tap's 1.8s mark.
    if showingComingSoon?.id == card.id {
        showingComingSoon = nil
    }
}
```
