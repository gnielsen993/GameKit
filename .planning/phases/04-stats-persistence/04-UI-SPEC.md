# Phase 04 — UI Design Contract

> Visual and interaction contract for the Stats & Persistence phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack note:** Native iOS 17+ / SwiftUI / Swift 6 project. **DesignKit** (local SPM at `../DesignKit`) fills the role shadcn fills on web; the shadcn / Tailwind preset format is structurally non-applicable. UI surfaces in P4 are **two existing screens** (`StatsView`, `SettingsView`) being extended — **not** new game-screen surfaces. P4 ships zero new DesignKit tokens (`gameNumber` already shipped in P3) and zero new generic components.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | DesignKit (local SPM dep at `../DesignKit`) — shadcn gate non-applicable for SwiftUI |
| Preset | Active preset is user-selected at runtime; **Forest** (Classic) is the first-launch default. P4 inherits the **same 6-preset audit set** locked in P3 (CLAUDE.md §8.12): `forest` (Classic) · `bubblegum` (Sweet) · `barbie` (Bright) · `cream` (Soft) · `dracula` (Moody) · `voltage` (Loud). Both StatsView and the new Settings rows must remain legible across all six presets in light + dark modes |
| Component library | DesignKit components (read-only consumption): `DKCard` (already wraps existing P1 stub stats sections) and the shared `settingsSectionHeader(...)` / `settingsNavRow(...)` `@ViewBuilder` helpers from `Screens/SettingsComponents.swift`. **No new generic components added in P4** — single-game/single-surface patterns stay local per CLAUDE.md §2 + §4 ("promote when proven — used in 2+ games") |
| Icon library | SF Symbols only — `square.and.arrow.up` (Export), `square.and.arrow.down` (Import), `trash` (Reset). No custom assets. SF Symbols are first-party Apple — not a third-party registry |
| Font | DesignKit `theme.typography` (system-defaulted): `title` (22pt semibold) · `headline` (17pt semibold) · `body` (17pt regular) · `caption` (12pt regular) · `monoNumber` (17pt monospaced) — **all sourced from `Theme.typography`, never declared inline**. P3-locked pattern: pair `theme.typography.monoNumber` with `.monospacedDigit()` modifier so digits do not jitter across stat updates |

**No new DesignKit tokens or components ship in P4.** The full token surface needed (radii `card`/`button`, spacing `xs..xxl`, typography `title`/`headline`/`body`/`caption`/`monoNumber`, color roles `surface`/`surfaceElevated`/`textPrimary`/`textSecondary`/`textTertiary`/`accentPrimary`/`danger`/`success`/`border`) is already on disk in DesignKit and consumed by P1/P3.

## Spacing Scale

Declared values (DesignKit `SpacingTokens` — already on disk at `../DesignKit/Sources/DesignKit/Layout/SpacingTokens.swift`):

| Token | Value | Usage in Phase 4 |
|-------|-------|------------------|
| `theme.spacing.xs` | 4pt | Section-header leading inset (matches existing `settingsSectionHeader` baseline); inner gap between difficulty label and its 4-stat row in a per-difficulty StatsView line |
| `theme.spacing.s` | 8pt | Vertical rhythm **between** the 3 difficulty rows inside the StatsView card; horizontal gap between SF Symbol glyph and label inside each Settings row (Export / Import / Reset); inner gap between column header row and the first data row in StatsView |
| `theme.spacing.m` | 12pt | Vertical inner padding of each Settings row (matches existing tappable row height target ≥ 44pt with `body` text); column gap inside StatsView grid (between games column → wins column → win % column → best-time column) |
| `theme.spacing.l` | 16pt | `DKCard` outer padding (already baked into `DKCard` — see DKCard.swift line 14) — applied to the StatsView "Minesweeper" card and any new Settings DKCard groupings; outer `VStack(spacing:)` between major Settings sections (already used by P1 stub — preserved) |
| `theme.spacing.xl` | 24pt | Vertical gap between "STATS" section header and the data card on StatsView; vertical gap between the existing "ABOUT" card and the new "DATA" group on SettingsView (so destructive Reset row reads as separated, not blurred into About) |
| `theme.spacing.xxl` | 32pt | Top safe-area inset before the first Settings section header on a fresh scroll position (only if needed — most sheets inherit from `NavigationStack` automatic top inset; do NOT add unless visible whitespace is too tight on iPhone SE) |

**Exceptions:**
- **Tappable Settings row minimum height = 44pt** (Apple HIG, matches `DKButton` floor). This is **not a spacing token** — it is an HIG-derived component-intrinsic dimension applied via `.frame(minHeight: 44)` directly on each row's outermost container. Per CLAUDE.md §1 / FOUND-07: bare `.frame(minHeight: 44)` is **not** flagged by the pre-commit hook (which scans for `padding(<int>)` / `cornerRadius:` / `Color(`); a minHeight literal is HIG-mandated and has no equivalent token).
- **No other exceptions.** P4 introduces zero ad-hoc spacing — all paddings/insets read `theme.spacing.{token}`.

**Hard cap:** Zero raw `.padding(8)` / `.padding(.horizontal, 16)` / `.padding(.vertical, 12)` integer literals in `Screens/StatsView.swift` or `Screens/SettingsView.swift`. Every padding call site reads `theme.spacing.{token}`. Verified by grep at phase-end (FOUND-07 hook covers this automatically on commit).

## Typography

| Role | DesignKit token | Used For | Notes |
|------|-----------------|----------|-------|
| Section heading | `theme.typography.caption` weight `.semibold` + `tracking(1.2)` + `theme.colors.textTertiary` | "STATS" / "DATA" / "ABOUT" group labels above each card group on Settings; "MINESWEEPER" group label above the StatsView card | **Reuses the existing `settingsSectionHeader(theme:_:)` helper from `Screens/SettingsComponents.swift`** (P1 pattern — do NOT redeclare). All-caps display is purely orthographic — the underlying string in xcstrings is in title case, `.textCase(.uppercase)` is applied at the view layer if needed |
| Card title | `theme.typography.title` (22pt semibold) | StatsView card sub-header **"Minesweeper"** above the 3 difficulty rows (D-25 — single source of game-scope inside the card) | Rendered inline at the top of the `DKCard` body, NOT via `DKSectionHeader` — keeps the card visually self-contained and avoids stacking two heading levels (the outer "MINESWEEPER" all-caps caption already covers the section semantic) |
| Row label | `theme.typography.body` (17pt regular) | Each difficulty name in the StatsView row ("Easy" / "Medium" / "Hard"); each Settings row label ("Export stats" / "Import stats" / "Reset stats"); column headers row inside the StatsView card ("Games" / "Wins" / "Win %" / "Best") if rendered | Default body for prose surfaces. Settings row label sits left-aligned with SF Symbol glyph leading and no trailing chevron (the row's destination is a sheet/alert, not a navigation push — per `settingsNavRow` precedent the chevron is reserved for nav destinations) |
| Stat numerals | `theme.typography.monoNumber` (17pt monospaced) **+ `.monospacedDigit()` modifier** | The 4 stat values per difficulty row: games count · wins count · win % · best time | **P3-locked pattern** — `monoNumber` alone is not enough; the `.monospacedDigit()` modifier is the load-bearing rule per `MinesweeperHeaderBar.swift` precedent. Required so digits do not jitter when stats update (`12 → 13 → 100`) and so the 4-column grid stays visually aligned across rows |
| Empty state body | `theme.typography.body` (17pt regular) + `theme.colors.textTertiary` | The single-line `"No games played yet."` copy that replaces the 3 difficulty rows when `minesRecords.isEmpty` (D-26 / SC2) | One line, centered horizontally inside the DKCard, calm tone — no icon, no illustration |
| Alert body | System default (`.alert(_:isPresented:actions:message:)` uses system styling — DesignKit tokens do not apply inside system alerts) | Reset confirmation alert body; schema-mismatch import alert body | System-rendered — outside DesignKit's reach. Strings come from `String(localized:)` per FOUND-05. No theming concern since the alert chrome inherits iOS conventions |

**Roles count:** 5 declared roles (section heading · card title · row label · stat numerals · empty state body), all already on the DesignKit typography surface. No new typography token needed.

**No `.font(.title2)` / `.font(.system(size: 18))` literals anywhere in `Screens/StatsView.swift` or `Screens/SettingsView.swift`.** Every `.font()` call site reads `theme.typography.{token}`. Verified by grep at phase-end.

**Heading line height:** SwiftUI's default for `theme.typography.title` and `theme.typography.body` (system-tuned ~1.2 / ~1.4). No explicit `lineSpacing` overrides in P4 — the existing P1 stub does not override and has shipped on 6 presets cleanly.

## Color

| Role | Token | Usage in Phase 4 |
|------|-------|------------------|
| Dominant (60%) | `theme.colors.background` | Stats scene root background; Settings scene root background (already applied via `.background(theme.colors.background.ignoresSafeArea())` in P1 stubs — preserved) |
| Secondary (30%) | `theme.colors.surface` (StatsView card body via `DKCard`; Settings card bodies via `DKCard`); `theme.colors.surfaceElevated` (NOT used in P4 — reserved for the Mines grid's unrevealed-vs-revealed cell hierarchy in P3) | The "card on background" hierarchy is conveyed by `surface` lifted off `background` — **not by tinting**. Both tokens already resolve correctly under all 6 preset categories |
| Accent (10%) | `theme.colors.accentPrimary` | **Not consumed in `Screens/StatsView.swift` or `Screens/SettingsView.swift` for P4.** No primary CTA on either screen — Export/Import/Reset rows are tappable rows (not `DKButton(.primary)`); the StatsView is read-only. Accent stays reserved for the Restart `DKButton(.primary)` inside the Mines game scene (locked P3). **Zero accent consumption in P4 surfaces** |
| Destructive (semantic) | `theme.colors.danger` | (a) **Reset stats row leading SF Symbol glyph tint** (`Image(systemName: "trash")` foreground) — telegraphs the destructive nature of the row before the user taps; (b) **Reset alert destructive button** ("Reset all stats") — the system `.alert(role: .destructive)` colors this automatically; (c) **Schema-mismatch import alert** title color (system handles this since the alert role is `.destructive` per D-21) |
| Success (semantic) | `theme.colors.success` | **Not consumed in P4.** Successful Export does NOT surface a green confirmation toast — per `04-CONTEXT.md` Deferred Ideas, the toast is a P5 polish item. Import success is implicit (the stats card refreshes via `@Query`) |
| Text primary | `theme.colors.textPrimary` | All non-destructive Settings row labels (Export / Import); StatsView card title ("Minesweeper"); StatsView row difficulty labels ("Easy" / "Medium" / "Hard"); StatsView stat numeric values |
| Text secondary | `theme.colors.textSecondary` | StatsView optional column-header row text ("Games" / "Wins" / "Win %" / "Best") if rendered as a header strip — softer than `textPrimary` so the data values pop. Settings About-card body copy (already P1) |
| Text tertiary | `theme.colors.textTertiary` | Section headers via `settingsSectionHeader` (P1 helper — preserved); empty state copy "No games played yet." (calmer than primary); the `"—"` placeholder rendered when a difficulty has zero records and best time is unknown (D-25) |
| Border | `theme.colors.border` | `DKCard` overlay stroke (already baked in — DKCard.swift line 19); horizontal divider line **between** the column-header row and the first data row inside the StatsView card (a subtle 1pt rule — `Rectangle().fill(theme.colors.border).frame(height: 1)`); horizontal divider lines **between** the 3 difficulty rows (same pattern) |

**Accent reserved-for list (explicit, exhaustive for P4 surfaces):**
- *(empty)* — zero accent consumption in `Screens/StatsView.swift` and `Screens/SettingsView.swift` in P4. Project-wide accent reservation (locked P3) remains: Mines Restart `DKButton(.primary)` background + Mines toolbar Menu open-state glyph tint.

**Destructive reserved-for list (explicit):**
1. Reset stats row leading SF Symbol (`trash` glyph) tint — `theme.colors.danger`
2. Reset confirmation alert destructive button ("Reset all stats") — system-tinted via `role: .destructive`
3. Import schema-mismatch error alert — system-tinted via `role: .destructive`

Three items. The Export glyph (`square.and.arrow.up`) and Import glyph (`square.and.arrow.down`) render on `theme.colors.textPrimary` — they are non-destructive transactions; only Reset earns the danger tint.

**Per-preset palette safety (THEME-01 / THEME-02 / A11Y-04 verification):**

| Preset audit set | Risk | Verification |
|------------------|------|--------------|

## Layout & Sizing

| Surface | Dimension | Justification |
|---------|-----------|---------------|
| StatsView root container | `ScrollView` containing `VStack(alignment: .leading, spacing: theme.spacing.l)` — preserves P1 stub baseline | Already validated on iPhone SE in P1 |
| StatsView card width | `frame(maxWidth: .infinity)` — full available width minus the outer `theme.spacing.l` screen-edge padding | Standard iOS settings/stats convention; 4 columns of monospaced numerals fit comfortably on 320pt iPhone SE width with `theme.spacing.m` (12pt) column gaps |
| StatsView column layout | `Grid(alignment: .leading, horizontalSpacing: theme.spacing.m, verticalSpacing: theme.spacing.s) { ... }` — one `GridRow` for the column-header strip, then 3 `GridRow`s for difficulties (or single `GridRow` for empty state). Difficulty label column is `.gridColumnAlignment(.leading)`; the 4 stat columns are `.gridColumnAlignment(.trailing)` so digits right-align cleanly | **Use SwiftUI `Grid` (iOS 16+)** — auto-aligns columns across rows without manual width math, handles Dynamic Type gracefully (P5 polish), and supersedes ad-hoc `HStack` width hacks. `LazyVGrid` would also work but is overkill for 3-4 rows of static data |
| StatsView column-header rule | 1pt horizontal `Rectangle().fill(theme.colors.border)` between header strip and first data row, spanning full grid width | Subtle delineation without a `Divider()` (which doesn't theme cleanly) |
| StatsView between-row rules | (none — `theme.spacing.s` vertical breathing room between rows is sufficient. Adding rules between data rows clutters the calm-minimal tone) | Decision per CLAUDE.md tonal direction — minimal chrome |
| StatsView card minimum height | Intrinsic — wraps to 3 rows + header strip natural height (~120pt). Empty state collapses to ~60pt | No explicit height constraint |
| Settings row height | `.frame(minHeight: 44)` — HIG minimum touch target | Apple HIG, matches `DKButton` floor (DKButton.swift line 33) |
| Settings row internal layout | `HStack(spacing: theme.spacing.s) { Image(systemName: glyph) Text(label) Spacer() }` — glyph leading, label adjacent, no trailing chevron (action rows are not nav rows) | Aligns SF Symbol and label on the same baseline; HStack `Spacer()` claims unused trailing width so the entire row is the tap target |
| Settings row tap target | `.contentShape(Rectangle())` on the outermost HStack so tapping anywhere in the 44pt strip triggers the action | Matches existing `settingsNavRow` pattern |

## Interaction Contracts

### Export flow
1. User taps "Export stats" row in Settings.
2. `isExporterPresented = true` → `.fileExporter` modal appears.
3. System renders document picker with `defaultFilename: "gamekit-stats-YYYY-MM-DD.json"` and `contentTypes: [.json]`.
4. User picks a destination → system writes the `Data` returned by `StatsExporter.export(modelContext:)` (D-16).
5. `onCompletion` closure: success path is silent (no toast — deferred to P5); failure path logs via `os_log` and does not surface a user alert (per `04-CONTEXT.md` Deferred — error toast is P5).

### Import flow
1. User taps "Import stats" row.
2. `isImporterPresented = true` → `.fileImporter` modal appears.
3. System renders document picker with `allowedContentTypes: [.json]`.
4. User picks a file → system passes `URL` to `onCompletion`.
5. App reads the file, calls `StatsExporter.importing(_:modelContext:)` (D-20).
6. **On success:** silent. The `@Query`-backed StatsView refreshes automatically.
7. **On `StatsImportError.schemaVersionMismatch`:** `isImportErrorAlertPresented = true` with schema-mismatch copy (D-21).
8. **On any other thrown error:** `isImportErrorAlertPresented = true` with generic import-error copy.

### Reset flow
1. User taps "Reset stats" row.
2. `isResetAlertPresented = true` → system `.alert(role: .destructive)` appears.
3. User taps **Cancel** → alert dismisses, no state change.
4. User taps **Reset all stats** → `GameStats.resetAll()` (D-13) wipes both `GameRecord` and `BestTime` in a single transaction → `try modelContext.save()` → alert dismisses → StatsView refreshes (now showing empty state).

### Mid-flow concerns
- **No undo affordance** for Reset — the alert is the only consent gate (per `04-CONTEXT.md` Deferred Ideas — 5-second snackbar undo is P5+ if ever).
- **No "Export first?" nudge** in the Reset alert (D-23) — the user can Export from the same Settings card one row above if they want a backup.
- **No conflict between concurrent flows** — only one of the three modals can be active at a time because the user can only tap one row at a time and `.alert`/`.fileExporter`/`.fileImporter` all dismiss before the next user interaction is possible.
