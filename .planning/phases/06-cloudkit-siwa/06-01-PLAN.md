<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App ↔ Keychain | Apple userID crosses from in-memory string into encrypted-at-rest secure enclave-backed storage. Untrusted reading from outside this boundary (e.g. another app, attached debugger on jailbroken device) is mitigated by `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` semantics. |
| Production target ↔ Test target | `InMemoryKeychainBackend` is a test-only artifact. If it leaks into the production target the AuthStore default-value EnvironmentKey would silently swap to in-memory in release builds (silent regression of SC2). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | `SystemKeychainBackend` | mitigate | Use `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` verbatim per CONTEXT line 202 + SC2; never fall back to `kSecAttrAccessibleAlways` or `kSecAttrAccessibleAfterFirstUnlock` (cross-device migration). Acceptance criteria asserts the literal token. |
| T-06-06 | Tampering | `kSecAttrService = "com.lauterstar.gamekit.auth"` | mitigate | Hardcode `static let serviceName = "com.lauterstar.gamekit.auth"` as a `let` constant; any drift breaks Keychain lookup but does not corrupt data. Smoke-tested in Plan 06-04 AuthStoreTests round-trip. |
| T-06-09 | Denial of Service | `SystemKeychainBackend.write` errSecDuplicateItem | mitigate | Idempotent write pattern — `try? delete(account:)` BEFORE `SecItemAdd` (RESEARCH lines 421-423). Avoids the entire duplicate-item failure class. |
| T-06-W-test-leak | Tampering / Information Disclosure | `InMemoryKeychainBackend.swift` placement | mitigate | File MUST live under `gamekit/gamekitTests/Helpers/` (test target only). Acceptance criteria greps the production target sources to confirm absence. |
</threat_model>
