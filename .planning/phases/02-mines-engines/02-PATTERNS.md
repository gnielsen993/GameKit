### NEW PATTERN: pure-Foundation engine struct/enum namespace

**No analog in repo** — P2 is the first engine phase. Reference: `.planning/research/ARCHITECTURE.md` "Component Responsibilities" table + "Pattern 1: Pure Engine + Observable ViewModel + Dumb View" §"Example" lines 178–219.

**Imports template (use verbatim across `BoardGenerator.swift`, `RevealEngine.swift`, `WinDetector.swift`):**
```swift
import Foundation
// NOTHING ELSE. No SwiftUI. No SwiftData. No GameplayKit. No UIKit.
// Engine purity is ROADMAP P2 SC5 — load-bearing test of this phase.
```

**Type-declaration shape (per ARCHITECTURE.md Component Responsibilities table — `enum` namespace of `static func`s):**
```swift
import Foundation

/// Pure board generation. Deterministic given an injected RNG.
/// Foundation-only — no SwiftUI/SwiftData/GameplayKit (ROADMAP P2 SC5).
public enum BoardGenerator {
    public static func generate(
        difficulty: MinesweeperDifficulty,
        firstTap: MinesweeperIndex,
        rng: inout some RandomNumberGenerator
    ) -> MinesweeperBoard {
        // 1. allCells = cartesian product of (rows × cols)
        // 2. safeZone = {firstTap} ∪ firstTap.neighbors8 (bounds-clamped)
        // 3. mineCandidates = allCells - safeZone
        // 4. mines = sample-without-replacement(mineCandidates, count: difficulty.mineCount)
        //    — single shot, no re-roll loop (PITFALLS.md Pitfall 1)
        // 5. precompute adjacency for every cell
        // 6. return immutable MinesweeperBoard
    }
}
```

**Recommendation:** Use `enum` (uninhabitable — no instance can be created) over `struct` for engine namespaces. ARCHITECTURE.md's "Component Responsibilities" table explicitly says `enum` namespace of `static func`s for `RevealEngine` and `WinDetector`, and `struct` for `BoardGenerator` — but `enum` is uniformly correct here since none of the three engines hold state. Planner discretion: pick one and apply uniformly across all three engine files (recommend `enum`).

**Visibility:** `internal` (default) is sufficient — these types are consumed by `MinesweeperViewModel` in the same module (P3). No `public` needed; `@testable import gamekit` reaches `internal`.

### NEW PATTERN: immutable value-type `Board` with copy-on-mutate

**No analog in repo.** Reference: `.planning/research/ARCHITECTURE.md` Component Responsibilities ("Pure struct...always return new board") + `02-CONTEXT.md` D-10 ("Board is an immutable value-type struct; engine functions return new Board, never mutate. inout mutation rejected — breaks SwiftUI @Observable snapshotting").

**Shape:**
```swift
import Foundation

public struct MinesweeperBoard: Equatable, Hashable, Codable {
    public let rows: Int
    public let cols: Int
    public let difficulty: MinesweeperDifficulty
    public let cells: [MinesweeperCell]   // flat [rows*cols] indexed row*cols + col
    public let mineCount: Int

    // Computed read accessors only — no mutating methods on Board.
    // Mutations land via engine functions returning a NEW Board:
    //   `let newBoard = RevealEngine.reveal(at: idx, on: board).board`
}
```

**Conformances to add:**
- `Equatable` — required by PITFALLS.md Pitfall 6 ("Make `Cell` `Equatable` so SwiftUI's diffing can skip unchanged cells"). Same applies to `Board` for VM-level diffing in P3.
- `Hashable` — useful for `Set`-based first-tap-safe exclusion (D-09: "Hashable enables Set-based first-tap-safe exclusion").
- `Codable` — D-02 ("stable serialization key for P4 stats / JSON export"); engine `Board` is `Codable`-trivial (composed of `Codable` value types) per `02-CONTEXT.md` `<deferred>` "Mid-game state persistence."

**Internal layout decision (per `02-CONTEXT.md` Claude's Discretion):** Flat `[Cell]` indexed by `row * cols + col` is recommended over nested `[[Cell]]` — Swift-idiomatic for fixed-size grids and marginally faster.

### NEW PATTERN: `MinesweeperIndex` Hashable struct

**No analog in repo.** Reference: D-09 ("`struct Index: Hashable { let row, col: Int }` across the entire engine API").

**Shape:**
```swift
import Foundation

/// A grid coordinate. Hashable so engine code can use Set-based exclusion
/// (`Set(allCells) - {tapped} - tapped.neighbors8`) per D-09 / PITFALLS.md
/// Pitfall 1.
public struct MinesweeperIndex: Hashable, Codable {
    public let row: Int
    public let col: Int

    /// Bounds-clamped 8-neighbors. Corner = 3 neighbors, edge = 5,
    /// interior = 8 (PITFALLS.md Pitfall 1: "bounds-clamp neighbors").
    public func neighbors8(rows: Int, cols: Int) -> [MinesweeperIndex] { ... }
}
```

**Critical:** `neighbors8(rows:cols:)` MUST clamp to board bounds. PITFALLS.md Pitfall 1 explicitly warns: "If exclusion logic uses '8 neighbors' without bounds-checking, off-board indices throw or get silently skipped, leaving a corner with fewer-than-expected exclusions."

### NEW PATTERN: `Cell.state` single-enum representation

**No analog in repo.** Reference: `02-CONTEXT.md` Claude's Discretion ("Recommend single enum — best self-documents in tests").

**Shape:**
```swift
import Foundation

public struct MinesweeperCell: Equatable, Hashable, Codable {
    public let isMine: Bool
    public let adjacentMineCount: Int   // 0…8, precomputed at generation (D-13 specifics)
    public var state: State

    public enum State: Equatable, Hashable, Codable {
        case hidden
        case revealed
        case flagged
        case mineHit       // terminal state — only the loss-triggering mine
    }
}
```

**Note on `revealed(adjacent: Int)`:** `02-CONTEXT.md` Claude's Discretion shows `.revealed(adjacent: Int)` as one option, but storing `adjacentMineCount` as a separate field on `Cell` (precomputed at generation per "Specific Ideas" → "Adjacency counts precomputed at board-generation time") is cleaner — `.revealed` enum case stays trivial, adjacency is read from the cell field whether revealed or not.

### NEW PATTERN: SplitMix64 deterministic test RNG

**No analog in repo.** Reference: D-12 ("custom SplitMix64 (~15 lines) in `gamekitTests/Helpers/SeededGenerator.swift`. Production engine target does NOT ship the seedable generator").

**Shape:**
```swift
import Foundation
@testable import gamekit

/// SplitMix64 PRNG. ~15 lines. Pure-Swift, deterministic, uniform, fast.
/// Test-only helper — not shipped in the production engine target (D-12).
struct SeededGenerator: RandomNumberGenerator {
    private var state: UInt64

    init(seed: UInt64) {
        self.state = seed
    }

    mutating func next() -> UInt64 {
        state &+= 0x9E3779B97F4A7C15
        var z = state
        z = (z ^ (z &>> 30)) &* 0xBF58476D1CE4E5B9
        z = (z ^ (z &>> 27)) &* 0x94D049BB133111EB
        return z ^ (z &>> 31)
    }
}
```

**Critical placement:** This file lives in `gamekit/gamekitTests/Helpers/SeededGenerator.swift` — under the **test target only**. If it ends up in the production app target, Engine purity is violated (`@testable import` is the seam).

**Why not `GameplayKit.GKMersenneTwisterRandomSource`?** D-12 explicitly rejects: "No GameplayKit dependency." STACK.md §"What NOT to Use" reinforces: "SpriteKit, RealityKit, GameplayKit ... bypasses DesignKit."

## Shared Patterns

### Foundation-only purity (applies to ALL 5 production engine + model files)

**Source:** ROADMAP P2 SC5 + ARCHITECTURE.md Component Responsibilities + STACK.md §1 ("Engines are pure value types, no actor at all... never touch ModelContext, never import SwiftUI") + PITFALLS.md "Anti-Pattern 1: ViewModel imports SwiftUI" + "Anti-Pattern 2: Engine touches ModelContext."

**Apply to:** `MinesweeperDifficulty.swift`, `MinesweeperIndex.swift`, `MinesweeperCell.swift`, `MinesweeperBoard.swift`, `MinesweeperGameState.swift`, `BoardGenerator.swift`, `RevealEngine.swift`, `WinDetector.swift`.

**Rule:** First (and ideally only) import statement is `import Foundation`. Any import of `SwiftUI`, `SwiftData`, `UIKit`, `GameplayKit`, `Observation`, or `Combine` in any of these files is a bug.

**Enforcement (planner suggestion):** Add a CI grep to fail the build if any file under `Games/Minesweeper/Engine/` matches `^import (SwiftUI|SwiftData|UIKit|GameplayKit|Observation|Combine)`. Cheap; protects the load-bearing invariant.

---

### File-size cap (CLAUDE.md §8.1, §8.5)

**Source:** CLAUDE.md ≤500-line hard cap, ≤400-line soft cap for views.

**Apply to:** All 9 P2 files.

**Expected sizes per `02-CONTEXT.md` "Established Patterns" / D-19:**
- `BoardGenerator.swift` ~80 lines
- `RevealEngine.swift` ~120 lines (incl. iterative flood-fill)
- `WinDetector.swift` ~30 lines
- Each model file ~30–80 lines
- Each test file <200 lines (D-19 explicit cap)
- `SeededGenerator.swift` ~15 lines (D-12 explicit)

**Trigger to split:** If `RevealEngine.swift` bloats past ~200 lines, split flood-fill into a sibling `FloodFill.swift` in the same `Engine/` folder (per `02-CONTEXT.md` "Established Patterns").

---

### Deterministic-by-seed testing

**Source:** D-11, D-12, D-13.

**Apply to:** All 3 test files.

**Rule:** Every randomized test passes `var rng = SeededGenerator(seed: <hardcoded>)` and feeds `&rng` to `BoardGenerator.generate`. No `SystemRandomNumberGenerator` in tests. Failure on seed N is reproducible by re-running with seed N.

---

### Iterative algorithms only (no recursion)

**Source:** ROADMAP P2 SC3 + PITFALLS.md "Performance Traps" → "Recursive flood-fill on Hard ... Stack overflow risk on degenerate boards."

**Apply to:** `RevealEngine.swift` flood-fill.

**Rule:** Use an explicit `Array<Index>` queue or stack. No `func floodFill(...) { ...; floodFill(neighbor) }` self-call. A test should assert this — e.g. by invoking flood-fill on a 16×30 board with mines clustered in one corner (worst-case ~470-cell cascade) and asserting it terminates.
