Variant 4: Warning + compromise

The Hard 16x30 board keeps its current cell size. A one-time banner tells the user "Video Mode works best with small PiP on Hard." The compact row applies the Compromise order from the plan-doc (hide time chip, collapse Settings into an overflow menu) to free a few points vertically. The user is invited to switch to PiP-small (which the 4-corner Dracula screenshots prove is fully playable at current cell size).

A11Y-05 / 06.1-03 deconfliction (ZERO RISK): no gesture or layout-engine changes. The existing MagnifyGesture + auto-scale cellSize system is untouched. Pinch is still available as a user-driven escape hatch if the user keeps PiP-large. This variant adds one banner UI element and one Compromise-order branch — nothing crosses the cell-gesture boundary.

Behavior

  • Cell size: unchanged — current 18pt floor preserved. Touch targets identical to non-Video-Mode play.
  • Banner: appears once per session when Hard + Large-top/Large-bottom PiP combination is detected. Pill-shaped, follows banner-placement rules from 08-BANNER-PLACEMENT.md. Dismissable; persists "dismissed" flag in-memory only.
  • Compromise order applied: step 4 (collapse Settings → overflow ⋯ menu); step 5 (hide time chip). Picker pill + flag/mine count + Back stay (critical info).
  • Squeeze accepted: the board still doesn't fit fully — bottom rows are clipped by the compact row. User can either: (a) switch to PiP-small via Control Center (the suggested action), or (b) pinch-zoom-out via existing A11Y-05.
  • A11Y-05 unchanged: pinch-zoom remains available as the user-driven escape hatch.

Pros

  • Zero gesture or layout-engine changes — lowest engineering risk.
  • Cell size + touch targets identical to v1.0 Hard play — zero legibility regression.
  • Existing A11Y-05 still available as escape hatch.
  • Honest with the user — concedes that Hard + Large PiP is a constrained case.
  • Ships in a single Phase 11 plan with no research-flag spike.

Cons

  • Concedes that Hard + Large PiP isn't a great experience — relies on user heeding the banner.
  • Adds one localized copy string (banner text); copy review needed.
  • If user ignores the banner AND doesn't pinch, the bottom-row squeeze persists for the whole game.
  • Banner display rules need precise wording: "once per session"? "once forever"? "every Hard+Large entry"? (Phase 11 plan locks this.)
